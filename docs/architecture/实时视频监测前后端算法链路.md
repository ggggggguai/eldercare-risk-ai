# 实时视频监测链路

## 完整流程

1. 萤石摄像头采集实时画面，并将视频上传到萤石开放平台。

2. 前端通过萤石提供的播放器或播放地址显示实时画面。

3. 用户开始监测后，前端通知业务后端。后端从萤石开放平台获取摄像头直播地址。

4. 后端通过 HTTP 调用算法容器的启动接口，把直播地址、设备编号、老人编号和结果回调地址传给算法。

5. 算法容器直接连接摄像头直播流，持续读取画面并进行风险分析。没有发现风险时，不发送风险事件。

6. 算法确认风险后，通过 HTTP 调用后端的风险结果接口，把风险等级、风险分数、置信度和判断理由发送给后端。

7. 后端保存风险事件，并通过 WebSocket 把结果实时推送给前端。

8. 前端收到风险事件后，在实时画面旁显示风险等级、风险分数和判断理由。

9. 用户停止监测时，前端通知后端，后端再调用算法容器的停止接口。算法停止拉流并释放资源。

## 链路摘要

```text
萤石摄像头
  -> 萤石开放平台
  -> 前端播放实时画面

前端请求开始监测
  -> 后端获取直播地址
  -> 后端通过 HTTP 启动算法
  -> 算法直接拉取并分析直播流
  -> 算法发现风险后通过 HTTP 回调后端
  -> 后端保存结果并通过 WebSocket 推送前端
  -> 前端显示风险结果
```

## 算法服务实际状态机

```text
starting -> running -> reconnecting -> running
starting/running/reconnecting -> stopping -> stopped
starting/running/reconnecting -> failed
```

单个服务进程最多保留一个非终止会话。地址更新会释放当前 `VideoCapture`，同一会话线程读取新地址，不创建第二个 YOLO 实例。停止信号通过可中断等待传递，最终释放视频连接、YOLO 引用和会话内 HTTP Client。

每帧仅执行一次 `YOLOv8-pose + ByteTrack`，同一 Ultralytics 结果同时适配出跟踪框与带相同 `track_id` 的姿态。主目标优先保持上一帧轨迹；丢失超过阈值后清空姿态窗口，再按最大人体框选择新轨迹。姿态质量、步态、坐站、近跌倒、个体基线和最终规则融合均通过内存对象衔接，不在实时主路径使用临时 JSONL。

当前疑似跌倒、倒地后静止、场景风险和活动节律均为工程规则 baseline，尚未经过临床标定。算法服务只输出标准化风险建议，业务后端负责持久化、消息通道和处置流程。
