## 2. 项目定位

本项目做的是“跌倒风险前置预警”，不是单纯的“跌倒动作检测”。

核心创新表达建议固定为：

```text
个体化行为基线 + 近跌倒事件检测 + 可解释分级预警
```

算法目标是从视频、姿态、行为节律和场景信息中识别老人跌倒风险变化，输出标准化风险事件 JSON。算法模块只负责输出风险等级、风险分数、置信度、风险因子和建议动作编码，不负责 App、后台、消息推送、工单流转、账号权限或线下处置。

必须避免的错误表述：

- 不要把当前模型描述成临床诊断模型。
- 不要把规则 baseline 描述成已完成医学验证的模型。
- 不要把 LE2I/IMViA 等公开跌倒数据集说成长期跌倒风险标签数据。
- 不要宣称已经完成实时萤石平台闭环，除非真实接入和验证已经完成。

## 3. 当前状态

事实：当前工程已经具备从部分结构化输入到风险事件 JSON 的可运行雏形，并且已拆出检测、姿态、质量控制、步态、坐站、近跌倒、个体基线和风险融合等层。

当前已实现：

| 层级 | 当前能力 | 主要文件 |
|---|---|---|
| 风险融合 | 从结构化特征输出 `AlgorithmEvent` 风险 JSON，当前为规则评分卡 | `src/elderly_monitoring/modules/fall_risk/pipeline.py` |
| 人体检测与跟踪 | 使用 Ultralytics YOLOv8 + ByteTrack 输出人体轨迹 JSONL | `src/elderly_monitoring/modules/fall_risk/tracking.py` |
| 姿态关键点 | 使用 YOLOv8 pose 输出关键点 JSONL，RTMPose 是后续适配方向 | `src/elderly_monitoring/modules/fall_risk/pose.py` |
| 姿态质量控制 | 关键点置信度过滤、短缺失插值、指数平滑、窗口质量摘要 | `src/elderly_monitoring/modules/fall_risk/pose_quality.py` |
| 步态稳定性 | 从 cleaned pose 中提取步态 proxy 和局部风险分 | `src/elderly_monitoring/modules/fall_risk/gait.py` |
| 坐站转换 | 从 cleaned pose 中识别候选坐站事件和局部风险分 | `src/elderly_monitoring/modules/fall_risk/sit_stand.py` |
| 近跌倒检测 | 基于姿态突变、快速下沉恢复、急停恢复、支撑 proxy 的规则 baseline | `src/elderly_monitoring/modules/fall_risk/near_fall.py` |
| 个体基线 | 使用滚动均值、标准差、分位数计算个人偏离分 | `src/elderly_monitoring/modules/fall_risk/baseline.py` |

计划或增强方向：

- RTMPose 适配器尚未替代当前 YOLOv8 pose 主入口。
- 实时视频流上的自动滑窗调度、事件合并和节流尚未形成完整主流程。
- Logistic Regression、LightGBM、TCN、ST-GCN、Transformer AutoEncoder、Risk Fusion Transformer 都属于后续训练或增强实验方向。
- 真实老人长期风险标签、盲测集、阈值校准、指标报告还需要补齐。

## 4. 固定算法路线

后续代码、文档、测试和汇报都必须对齐这条主线：

```text
萤石设备或开放平台视频流
  ↓
人体检测与跟踪
  ↓
人体姿态关键点提取
  ↓
关键点质量控制与时序平滑
  ↓
步态稳定性分析
  ↓
坐站转换能力分析
  ↓
近跌倒事件检测
  ↓
个体化行为基线建模
  ↓
轻量风险融合模型 + 规则校准
  ↓
跌倒风险等级 + 置信度 + 可解释风险因子 + 预警动作建议
```

这条路线有三个硬约束：

1. 不能从原始视频直接跳到最终风险结论。即使某层暂时是规则 baseline，也要保留该层的输入输出和文档说明。
2. 局部模块只输出局部分数和证据，不直接输出最终 `risk_level` 或业务动作。
3. 最终对外只通过统一 `AlgorithmEvent` schema 输出，不为某个模块临时新增一套对外接口。

## 5. 分层职责和输入输出

### 5.1 人体检测与跟踪

职责：

- 检测老人位置。
- 维持跨帧 `track_id`。
- 输出人体框、中心点、速度、场景区域和跟踪置信度。

输入：

```text
视频帧或离线视频文件
```

输出：

```text
tracks JSONL
```

注意事项：

- `track_id` 是跟踪维度，不等同于长期 `person_id`。
- 多人场景下需要明确主目标选择策略，不能默认所有人都是被监测老人。
- 跟踪中断或 ID 切换会影响速度、步态和基线，必须在输出质量或下游逻辑中体现。

### 5.2 姿态关键点提取

职责：

- 将人体框或视频帧转换为结构化骨架关键点。
- 输出肩、髋、膝、踝、腕等关键点坐标和置信度。

当前工程后端：

```text
YOLOv8 pose
```

计划后端：

```text
RTMPose
```

注意事项：

- 当前默认使用 0-1 归一化图像坐标，不能直接解释为真实世界距离、速度或角度。
- 不要在姿态提取层输出跌倒风险等级。
- 后续接 RTMPose 时，应保持 JSONL 字段兼容，不能迫使下游模块整体重写。

### 5.3 关键点质量控制与时序平滑

职责：

- 标记低置信度、缺失和异常跳变关键点。
- 对短缺失片段做有限插值。
- 输出 `x_smooth`、`y_smooth` 和窗口质量摘要。

下游必须优先使用：

```text
x_smooth / y_smooth
valid
is_jump_outlier
window_quality
```

注意事项：

- 不要覆盖原始 `x` / `y`，原始观测必须保留。
- 低质量窗口不能被当成高风险窗口。质量不足应该降低置信度或触发 `manual_review`，不是制造风险分。
- 步态、坐站和近跌倒模块都应先检查 `window_quality` 中对应的可用性标记。

### 5.4 步态稳定性分析

职责：

- 从 cleaned pose 提取步态稳定性 proxy。
- 输出 `gait_risk_score` 和可解释 `risk_factors`。

当前特征方向：

- 中心点速度均值和变异系数。
- 左右踝相对髋中心运动幅度。
- 左右下肢运动不对称性。
- 髋部横向摆动。
- 停顿、拖步、小碎步 proxy。

注意事项：

- `gait_risk_score` 是局部分数，不是最终跌倒风险等级。
- 当前没有真实步态风险标签校准，阈值调整必须通过验证集和误报分析支撑。
- 摄像头视角、下肢出画和遮挡会显著影响该模块。

### 5.5 坐站转换能力分析

职责：

- 检测候选坐下、起身或未知坐站转换事件。
- 输出 `sit_stand_risk_score`、耗时、失败尝试 proxy、起身后摇晃和疑似支撑使用。

注意事项：

- 腕部关键点只提供支撑使用弱证据，不能单独判定老人扶了墙或家具。
- 起身耗时和躯干角都是 2D proxy，不是医学量表。
- 低质量窗口必须输出质量不足因子并限制风险分，不能直接判高风险。

### 5.6 近跌倒事件检测

职责：

- 检测踉跄、横向失衡、快速下沉恢复、急停恢复、疑似支撑接触等近跌倒线索。
- 输出 `near_fall_event_score`、`event_type`、结构化 `evidence` 和 `risk_factors`。

注意事项：

- 近跌倒是本项目的关键创新点，也是最容易误报的模块。
- 正常转身、绕障、下蹲、挥手、坐下、镜头抖动都可能触发近跌倒 proxy。
- `support_contact_proxy` 只能作为弱证据，不能单独强判。
- 高分近跌倒应由融合层结合步态、坐站、基线、场景和质量信息再决定风险等级。

### 5.7 个体化行为基线建模

职责：

- 按 `person_id` 建立个人历史统计。
- 对当前窗口计算相对个人历史的偏离程度。
- 输出 `baseline_deviation_score`、`baseline_reference`、`baseline_quality` 和 `deviation_factors`。

注意事项：

- `track_id` 不能作为长期人员 ID。
- 历史样本不足时应输出 `insufficient_baseline_history`，并限制偏离分上限。
- 个体基线冷启动阶段需要群体阈值或保守策略，不能假装已经有稳定个人基线。

### 5.8 风险融合

职责：

- 融合步态、坐站、近跌倒、个体基线、场景、活动节律、疑似跌倒和长时间静止等信号。
- 输出最终 `risk_level`、`risk_score`、`confidence`、`trigger_event`、`risk_factors` 和 `recommended_action`。

当前主线：

```text
规则评分卡
```

后续主线候选：

```text
规则评分卡 + Logistic Regression / LightGBM
```

增强实验：

```text
Risk Fusion Transformer
```

注意事项：

- 融合层是唯一应该输出最终风险等级的层。
- 如果修改分数权重、等级阈值或动作建议映射，必须同步更新测试、README 和报告说明。
- `risk_factors` 必须保留可解释性，不能只输出模型黑盒分数。

## 6. 风险等级和建议动作

风险等级：

| 等级 | 含义 | 典型处理 |
|---:|---|---|
| 0 | 正常 | 仅记录或不展示异常 |
| 1 | 低风险 | 持续观察 |
| 2 | 中风险 | 建议提醒老人或家属关注 |
| 3 | 高风险 | 建议主动通知家属 |
| 4 | 紧急风险 | 疑似跌倒或长时间倒地，建议紧急告警 |

建议动作编码：

| 编码 | 含义 |
|---|---|
| `record_only` | 仅记录 |
| `observe` | 持续观察 |
| `remind_user` | 建议提醒老人 |
| `notify_guardian` | 建议通知家属 |
| `emergency_alert` | 建议紧急告警 |
| `manual_review` | 建议人工复核 |

注意：算法只输出建议动作编码，不直接发消息、不拨电话、不创建工单。

## 8. 开发注意事项

### 8.1 分数语义

局部分数：

```text
gait_risk_score
sit_stand_risk_score
near_fall_event_score
baseline_deviation_score
scene_risk_score
activity_rhythm_score
```

这些分数只能表示局部风险线索，不能单独对外解释为最终跌倒风险等级。

最终分数：

```text
risk_score
risk_level
confidence
recommended_action
```

这些字段只能由融合层输出。

### 8.2 质量优先于风险

低质量输入的正确处理方式：

- 标记质量不足。
- 降低置信度。
- 限制局部分数上限。
- 必要时建议人工复核。

错误处理方式：

- 把缺失点或跳变点直接解释为老人异常。
- 因为关键点质量差而提高风险等级。
- 在没有证据窗口的情况下输出高风险事件。

### 8.3 字段兼容

新增字段应遵守：

- 能兼容旧字段就不要改名。
- 必须改名时，更新 README、接口文档、样例 JSON 和测试。
- 所有输出都要包含足够的元数据，至少包括 `person_id`、时间字段、场景字段或质量字段。

### 8.4 数据和隐私

使用真实或半真实老人数据时：

- 使用脱敏 `person_id`，不要保存姓名、手机号、身份证号、住址等可识别身份信息。
- 保存授权记录、数据来源、采集时间范围和退出删除机制。
- 训练、验证、测试应按人员划分，避免同一老人同时出现在训练和测试中。
- 日志和样例文件不要写入个人身份信息。

### 8.5 评价必须有负样本

跌倒风险模型很容易只展示成功案例。比赛和工程评审都会关心误报，所以必须保留：

- 正常白天行走。
- 正常坐下和起身。
- 正常转身。
- 正常下蹲拿东西。
- 家人或其他人进入画面。
- 遮挡、低光、下肢出画、摄像头抖动。

没有这些负样本，就不能证明近跌倒检测真的可靠。

## 9. 优化方向

### 9.1 近期工程优化

优先级最高的是把当前 baseline 做稳，而不是马上堆复杂模型。

建议任务：

1. 完成视频到风险事件的端到端离线演示脚本。
2. 为每个下游模块补充误报案例表，记录触发窗口、关键点质量、风险因子和人工判断。
3. 增加事件合并和节流策略，避免同一事件连续输出多条高风险 JSON。
4. 增加场景矩阵测试，覆盖白天、夜间、正面、侧面、遮挡、多人和家具遮挡。
5. 将阈值集中到配置文件或参数对象，避免散落在多个模块中难以复现实验。
6. 为低质量窗口建立统一降权策略，避免不同模块各自处理造成冲突。

### 9.2 姿态和跟踪优化

可做方向：

- 接入 RTMPose 适配器，并保持当前 JSONL 契约不变。
- 比较 YOLOv8 pose 与 RTMPose 在居家低光、侧身、下肢遮挡场景下的关键点质量。
- 增强 ID 切换检测，给下游输出 track continuity 质量因子。
- 对常见安装角度建立场景配置，例如客厅、床边、走廊、卫生间门口。

评价重点：

- 有效关键点帧比例。
- 核心关键点覆盖率。
- ID switch 次数。
- 下游步态和近跌倒误报变化。

### 9.3 步态和坐站优化

可做方向：

- 将规则分数组件拆成可配置权重，便于阈值搜索。
- 增加转身稳定性、路径偏移、起身后稳定时间等特征。
- 用少量标注数据训练 Logistic Regression 或 LightGBM，替代纯规则权重。
- 后续数据充足时再尝试轻量 TCN。

评价重点：

- 坐站困难召回率。
- 正常起身误报率。
- 步态异常 F1。
- 不同摄像头角度下的稳定性。

### 9.4 近跌倒检测优化

可做方向：

- 先做规则阈值扫描和误报归因，再上异常检测模型。
- 使用 One-Class SVM、Isolation Forest 或局部离群因子做小样本异常检测对比。
- 数据足够后再尝试 Transformer AutoEncoder。
- 引入事件确认窗口，要求多个证据同时出现才提升到强近跌倒。

评价重点：

- 近跌倒事件召回率。
- 误报率。
- 平均提前预警时间。
- 正常转身、下蹲、绕障和挥手的误报。

### 9.5 个体化基线优化

可做方向：

- 明确冷启动阶段策略：无基线时使用群体阈值和保守置信度。
- 区分短期波动和长期趋势，例如小时级、天级、周级。
- 对活动节律、夜间活动、场景分布做按时间段建模。
- 对低质量历史样本降权，避免污染个人基线。

评价重点：

- 使用个体基线前后的高风险召回率和误报率变化。
- 冷启动阶段误报。
- 长期趋势预警案例数量。

### 9.6 风险融合优化

可做方向：

- 保留当前规则评分卡作为可复现 baseline。
- 在有标签后训练 Logistic Regression 或 LightGBM，并与规则 baseline 做对比。
- 用 SHAP、特征贡献或规则命中解释保留 `risk_factors`。
- 风险等级输出要做稳定性约束，避免 1 秒内频繁上下跳变。

评价重点：

- 高风险召回率。
- 3 级及以上误报率。
- F1、AUC、分级一致性。
- 风险等级变化稳定性。
- 事件输出延迟。

## 10. 挑战杯材料视角

比赛材料里建议突出三件事：

1. 从事后识别转向前置预警：不只识别已经摔倒，还识别跌倒前的失衡、坐站困难和行为退化。
2. 个体化：同样的步速下降对不同老人意义不同，要和个人历史基线比较。
3. 可解释：输出风险等级的同时给出风险因子和证据时间窗，便于家属、社区或评审复核。

需要准备的证据：

- 算法流程图。
- 风险事件 JSON 样例。
- 典型事件时间轴。
- 误报和漏报案例表。
- 消融实验表：规则阈值、无个体基线、无近跌倒检测、完整模型。
- 指标表：Accuracy、Precision、Recall、F1、误报率、漏报率、提前预警时间、推理延迟。

不要只放模型名字。评审更关心是否能运行、是否可解释、是否有真实或半真实场景验证。

## 12. 合并前检查清单

改代码前：

- 明确自己修改的是哪一层。
- 明确输入字段、输出字段和下游消费者。
- 看过对应测试和 README 描述。

改代码时：

- 不跳过固定算法路线中的中间层。
- 不把局部分数改成最终风险等级。
- 不因为质量差直接判高风险。
- 不破坏 `AlgorithmEvent` schema。
- 不在样例、日志或测试数据中写入真实身份信息。

提交前：

- 跑过相关窄范围测试。
- 涉及检测、姿态或跟踪时跑过真实视频烟测。
- 修改字段、阈值或行为时同步更新文档和样例。
- 记录已知限制，不把未验证能力写成已完成能力。

## 13. 当前最重要的风险

当前项目的主要风险不是“模型不够复杂”，而是下面几件事还需要证据支撑：

- 近跌倒定义和阈值是否能在正常动作中控制误报。
- 个人基线是否有足够历史数据和冷启动策略。
- 公开数据集是否能支撑比赛所需的真实场景说服力。
- 事件 JSON 是否能与总系统或演示链路稳定对接。
- 轻量主线是否能在现场演示中保持实时、稳定和可解释。

后续优化应优先补这些证据，再决定是否引入更复杂的深度模型。
