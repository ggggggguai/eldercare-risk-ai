# 第 7 步：近跌倒事件检测执行计划

## 1. 背景与当前位置

### 事实

- 固定算法路线当前已经推进到“坐站转换能力分析” baseline。
- 当前已存在并可作为上游输入的模块包括：
  - `src/elderly_monitoring/modules/fall_risk/tracking.py`
  - `src/elderly_monitoring/modules/fall_risk/pose.py`
  - `src/elderly_monitoring/modules/fall_risk/pose_quality.py`
  - `src/elderly_monitoring/modules/fall_risk/gait.py`
  - `src/elderly_monitoring/modules/fall_risk/sit_stand.py`
  - `src/elderly_monitoring/modules/fall_risk/pipeline.py`
- `pose_quality.py` 已输出下游可复用的清洗、平滑和质量字段，包括 `x_smooth/y_smooth`、`valid`、`is_jump_outlier`、`core_keypoint_quality` 和 `window_quality.usable_for_near_fall`。
- `gait.py` 已实现步态稳定性窗口级规则 baseline，输出 `gait_risk_score`、`gait_stability_features`、`quality_coverage` 和 `risk_factors`。
- `sit_stand.py` 已实现坐站转换候选事件级规则 baseline，输出 `sit_stand_risk_score`、`sit_stand_features`、`quality_coverage` 和 `risk_factors`。
- `FallRiskPipeline` 已经可以消费结构化输入字段 `near_fall_event_score`，并在 `near_fall_event_score >= 0.70` 时触发高风险融合逻辑。

### 需要明确的边界

`pipeline.py` 里已有 `near_fall_event_score` 输入字段，但这不等于近跌倒事件检测模块已经实现。

当前缺口是：

- 尚无独立 `near_fall.py`。
- 尚无近跌倒事件 JSONL 生成脚本。
- 尚无近跌倒事件单元测试。
- 尚无从 cleaned/smoothed pose JSONL 中提取近跌倒候选事件、特征、分数和证据的实现。

本计划对应固定算法路线第 7 步：

```text
坐站转换能力分析
  -> 近跌倒事件检测
  -> 个体化行为基线建模
```

本阶段只写执行计划，不实现代码。

## 2. 模块目标

### 近跌倒事件定义

近跌倒事件指“出现明显失衡、快速下沉、踉跄、急停、疑似扶物恢复等跌倒前置动作，但人体没有进入稳定倒地状态，且随后恢复到站立、行走或可持续活动状态”的短时事件。

典型例子：

- 行走中身体中心突然横向偏移，随后恢复。
- 髋部或躯干快速下沉，但没有持续倒地。
- 行走节律中断、急停，随后恢复移动。
- 手腕快速移向身体侧边或画面边缘并保持，疑似扶墙、扶家具或扶手恢复。
- 异常下蹲或半蹲后快速恢复。

### 与真正跌倒事件的区别

近跌倒事件不是“已经跌倒”：

| 类型 | 关键区别 |
|---|---|
| 近跌倒事件 | 有失衡或恢复动作证据，但未确认身体倒地、长时间静止或需要紧急跌倒后响应 |
| 真正跌倒事件 | 身体姿态进入倒地状态，或出现跌倒后长时间静止等强触发证据 |

因此第一版近跌倒模块不输出 `fall_event_score`，不替代后续真正跌倒检测或长时间静止检测，也不把近跌倒和真正跌倒混成一个模块。

### 第一版目标

第一版目标是可解释规则/统计 baseline：

- 从清洗后的 2D 姿态序列中识别短时异常候选片段。
- 提取可复核的近跌倒 proxy 特征。
- 输出局部 `near_fall_event_score`、事件证据、质量覆盖和风险因子。
- 为 `FallRiskPipeline` 提供可消费的归一化近跌倒事件分数。

第一版不做临床判断，不声称能诊断跌倒风险，也不直接输出最终 `risk_level`。

### 模块输出不包括

近跌倒模块输出的是局部事件结果，而不是最终融合结果：

- 输出 `near_fall_event_score`。
- 输出 `event_type`。
- 输出 `near_fall_features`。
- 输出 `quality_coverage`。
- 输出 `risk_factors`。
- 不输出最终 `risk_level`。
- 不输出最终 `recommended_action`。

最终整体跌倒风险等级仍由 `FallRiskPipeline` 或后续融合层结合步态、坐站、近跌倒、个体基线、场景和活动节律共同判断。

## 3. 输入契约

### 输入来源

输入来自 `pose_quality.py` 输出的 cleaned/smoothed pose JSONL。每行表示一人一帧，已按姿态质量控制层补充关键点有效性、平滑坐标、异常跳变和窗口质量摘要。

建议输入路径形态：

```text
data/processed/fall_risk/poses/*_poses_cleaned.jsonl
```

### 必须优先使用的字段

近跌倒模块必须优先使用以下字段：

| 字段 | 使用要求 |
|---|---|
| `keypoints[].x_smooth` / `keypoints[].y_smooth` | 优先用于速度、加速度、位移、路径偏移和角度变化计算 |
| `keypoints[].valid` | 仅把有效关键点作为可靠观测 |
| `keypoints[].is_jump_outlier` | 跳变点不直接删除，但应降权或排除出强触发证据 |
| `core_keypoint_quality` | 用于单帧质量加权和窗口质量统计 |
| `window_quality.usable_for_near_fall` | 用于近跌倒质量门控；低质量窗口不能被误判为高风险 |

可复用的核心关键点：

```text
left_shoulder
right_shoulder
left_hip
right_hip
left_knee
right_knee
left_ankle
right_ankle
```

可选增强关键点：

```text
left_wrist
right_wrist
```

腕部关键点只用于“疑似支撑接触”的弱 proxy。腕部缺失不应阻断近跌倒基础事件识别。

### 可选结合的上游特征

第一版可以结合 gait/sit_stand 的窗口或事件特征作为弱上下文，例如：

- `gait_stability_features.pause_frame_ratio`
- `gait_stability_features.center_speed_cv`
- `gait_stability_features.hip_lateral_sway`
- `sit_stand_features.post_stand_sway_component`
- `sit_stand_features.support_usage_component`
- `sit_stand_risk_score`

但第一版不要强依赖 gait/sit_stand 输出。原因：

- 近跌倒事件应能只从 cleaned pose JSONL 独立运行。
- 步态窗口、坐站事件和近跌倒候选事件的时间尺度不同。
- 强依赖上游局部分数会让误差级联，降低可测试性。

建议实现时把 gait/sit_stand 作为可选输入或后续批处理融合上下文，而不是 `near_fall.py` 的必需参数。

### 建议输入 JSONL 示例

## 4. 输出契约

输出为近跌倒事件级 JSONL。每条记录表示一个近跌倒候选事件或一个因质量不足而无法判断的事件窗口。

建议输出路径形态：

```text
data/processed/fall_risk/features/*_near_fall.jsonl
```

### 字段表

| 字段 | 类型 | 必需 | 含义 |
|---|---|---:|---|
| `person_id` | string | 是 | 人员 ID，来自上游姿态记录 |
| `track_id` | string/int/null | 是 | 跟踪 ID，来自上游姿态记录 |
| `scene_region` | string/null | 是 | 场景区域，如 `home`、`bedroom_bedside` |
| `start_time` | number | 是 | 事件开始时间，单位秒 |
| `end_time` | number | 是 | 事件结束时间，单位秒 |
| `near_fall_event_score` | number | 是 | 0-1 近跌倒局部分数，越高表示近跌倒线索越强 |
| `event_type` | string | 是 | 近跌倒事件类型枚举 |
| `near_fall_features` | object | 是 | 中间特征和评分组件，用于调参、复核和报告 |
| `quality_coverage` | object | 是 | 有效帧比例、核心点覆盖率、插值比例、跳变数量和质量不足标记 |
| `risk_factors` | array[string] | 是 | 可解释风险因子，建议用机器可读 code |
| `model_version` | string | 是 | 第一版建议 `near-fall-rule-v0.1` |

可选字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `event_id` | string | 事件唯一 ID，可由 `person_id + track_id + start_time + end_time` 生成 |
| `frame_start` | int/null | 起始帧 ID |
| `frame_end` | int/null | 结束帧 ID |
| `evidence` | array[object] | 结构化证据片段，如触发项、峰值时间、峰值数值 |

### `event_type` 枚举

第一版必须支持：

```text
stumble_or_lateral_loss
rapid_descent_recovery
sudden_stop_recovery
support_contact_proxy
abnormal_crouch_recovery
unknown_near_fall
```

质量不足时建议仍输出：

```text
unknown_near_fall
```

并设置：

```json
{
  "near_fall_event_score": 0.0,
  "quality_coverage": {
    "insufficient_near_fall_quality": true
  },
  "risk_factors": [
    "insufficient_near_fall_quality"
  ]
}
```

低质量窗口不能通过“异常抖动”获得高分。

## 5. 事件类型设计

### `stumble_or_lateral_loss`

含义：疑似踉跄、侧向失衡或短时重心偏移后恢复。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 身体中心或髋中心横向速度、加速度突增；路径相对前后移动方向出现明显侧向偏移；随后回到原有运动轨迹或站立姿态 |
| 关键点依据 | 左右髋中心、左右肩中心；可选使用踝部运动判断是否仍在行走 |
| 误判风险 | 跟踪 ID 切换、相机抖动、老人正常转身、侧向绕障、低质量关键点跳变 |

### `rapid_descent_recovery`

含义：疑似身体快速下沉但随后恢复，没有形成倒地或长时间静止。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 髋中心或身体中心 `y_smooth` 短时间明显增大，随后回到接近原高度；恢复期未进入倒地静止状态 |
| 关键点依据 | 左右髋、左右肩；可选使用膝、踝判断是否为坐下或正常下蹲 |
| 误判风险 | 正常坐下、弯腰、捡东西、镜头俯视导致垂直变化压缩、下肢遮挡 |

### `sudden_stop_recovery`

含义：行走中突然急停或明显节律中断，随后恢复。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 身体中心速度从正常移动快速降到低速，持续短时间后恢复；可能伴随横向晃动或躯干角变化 |
| 关键点依据 | 髋中心、肩中心、脚踝相对髋中心运动；可选结合步态窗口中的停顿/犹豫 proxy |
| 误判风险 | 正常停下、等待、转身准备、与他人交互、画面边缘出入导致速度变化 |

### `support_contact_proxy`

含义：疑似通过墙面、家具、扶手等支撑物恢复，但没有真实接触标签。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 单侧或双侧腕部快速移向身体侧边、画面边缘或固定区域后短暂保持；同时身体中心发生异常偏移或下沉 |
| 关键点依据 | 左右腕、左右肩、左右髋；腕部相对身体中心的位置和速度 |
| 误判风险 | 腕部关键点误检、挥手、拿东西、身体遮挡、家具不可见、画面边缘并不等于真实墙面 |

注意：`support_contact_proxy` 只能作为弱证据，不能单独强判高分。

### `abnormal_crouch_recovery`

含义：疑似异常下蹲、半蹲或膝髋快速弯曲后恢复。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 髋部快速下移，膝/踝相对关系显示下肢弯曲增强，随后恢复；持续时间短于正常坐下或坐站转换 |
| 关键点依据 | 髋、膝、踝，辅以肩髋中心关系 |
| 误判风险 | 正常蹲下捡物、坐下动作、坐站转换、低角度摄像头导致腿部遮挡 |

### `unknown_near_fall`

含义：存在局部异常证据，但无法稳定归入以上类型，或窗口质量不足无法判断。

| 维度 | 说明 |
|---|---|
| 可观测信号 | 轻微多证据组合、事件边界不完整、类型冲突或质量不足 |
| 关键点依据 | 根据可用核心点计算最小特征集 |
| 误判风险 | 容易混入动作不完整、遮挡、出画、正常活动变化；应保守给分 |

## 6. 特征设计

### 基础几何量

每帧建议从 cleaned pose 中计算：

- 肩中心：左右肩中点。
- 髋中心：左右髋中点。
- 膝中心：左右膝中点。
- 踝中心：左右踝中点。
- 身体中心：肩中心与髋中心加权中心。
- 躯干长度 proxy：肩中心到髋中心距离。
- 髋宽/肩宽 proxy：用于尺度归一化。
- 躯干角 proxy：肩中心到髋中心向量相对图像竖直方向的夹角。

只使用满足以下条件的点：

- `valid == true`
- `x_smooth/y_smooth` 可用
- `is_jump_outlier != true`，或仅作为弱证据并降权
- 当前帧或所在窗口满足 `window_quality.usable_for_near_fall`

### 可提取特征

第一版可从 2D 姿态序列中提取以下特征：

| 特征 | 说明 |
|---|---|
| 髋部/躯干横向速度与加速度突变 | 计算身体中心、髋中心或肩中心的 `x_smooth` 一阶/二阶差分，识别侧向失衡 |
| 髋部快速下沉后恢复 | 检测髋中心 `y_smooth` 快速增大后回落，要求恢复段存在 |
| 身体中心路径偏移 | 计算身体中心相对起止连线的最大偏离或均值偏离 |
| 躯干角度突变 | 计算躯干角变化幅度和角速度峰值 |
| 急停与恢复 | 检测中心速度快速下降到阈值以下后恢复到移动状态 |
| 手腕快速接近身体侧边/画面边缘作为支撑 proxy | 检查腕部相对身体中心的快速移动和短时稳定 |
| 低质量窗口降权 | 根据可用帧比例、核心点覆盖率、插值比例和跳变数量降低分数 |

### 重要限制

这些特征都是归一化图像坐标 proxy，不能直接解释为真实世界物理量：

- 横向速度不是真实米/秒。
- 加速度不是真实米/秒平方。
- 髋部下沉不是真实高度变化。
- 躯干角不是三维身体姿态角。
- 手腕接近画面边缘不等于真实扶墙或扶家具。

文档、日志、报告和展示层都应避免把这些 proxy 写成临床或物理确定结论。

## 7. 评分规则 baseline

第一版建议使用透明评分卡，而不是直接训练复杂时序模型。规则分数应稳定、可调参、可单元测试、可解释。

### 候选事件切分

建议按 `person_id + track_id` 分组并排序后，使用短滑窗生成候选片段：

| 参数 | 建议初值 | 说明 |
|---|---:|---|
| `window_sec` | `1.0 - 2.0` | 近跌倒通常是短时事件 |
| `step_sec` | `0.25 - 0.5` | 允许滚动检测 |
| `min_event_frames` | `5` | 低于该帧数不强判 |
| `merge_gap_sec` | `0.5` | 相邻同类候选可合并 |

无可靠时间戳时可使用固定帧数窗口，但输出仍应保留 `start_time/end_time` 的 fallback。

### 评分卡建议

建议分数组件：

```text
near_fall_event_score =
  0.25 * lateral_instability_component
  + 0.20 * rapid_descent_recovery_component
  + 0.15 * sudden_stop_recovery_component
  + 0.10 * support_contact_component
  + 0.15 * abnormal_crouch_component
  + 0.15 * trunk_angle_change_component
```

然后应用质量调整：

```text
adjusted_score = raw_score * quality_multiplier
```

其中：

```text
quality_multiplier = clamp(0.4 + 0.6 * usable_frame_ratio - quality_penalty, 0.0, 1.0)
```

质量惩罚可来自：

- 核心点覆盖率低。
- 插值比例高。
- 跳变点数量多。
- `window_quality.usable_for_near_fall == false` 的帧比例高。
- 连续低质量帧比例高。

### 强触发与弱触发

强触发建议必须满足“异常突变 + 恢复证据 + 质量可用”：

- 明显横向失衡，并在短窗口内恢复。
- 明显快速下沉，并在短窗口内恢复。
- 急停恢复，同时伴随横向偏移或躯干角突变。

弱触发只能升分，不能单独强判：

- 腕部支撑 proxy。
- 单帧躯干角突变。
- 单帧中心速度突增。
- 轻微路径偏移。
- 质量覆盖偏低。

第一版必须避免“一个坏关键点跳变等于近跌倒高风险”。

### 低质量处理

低质量窗口处理原则：

- 当 `window_quality.usable_for_near_fall` 长时间为 false、核心点覆盖率不足或有效帧太少时，输出 `near_fall_event_score = 0.0` 或保守低分。
- `risk_factors` 标记 `insufficient_near_fall_quality`。
- `quality_coverage.insufficient_near_fall_quality = true`。
- 不把遮挡、出画、跳变和低置信度误判成高风险。

### 局部分数解释区间

建议第一版局部解释：

| `near_fall_event_score` | 局部含义 |
|---:|---|
| `< 0.25` | 未见明显近跌倒事件 |
| `0.25 - 0.50` | 轻微信号或弱证据，需要观察 |
| `0.50 - 0.70` | 疑似近跌倒事件，建议融合层关注 |
| `>= 0.70` | 近跌倒强证据，融合层可触发高风险逻辑 |

上述区间不是最终跌倒风险等级。最终 `risk_level`、`risk_score`、`confidence` 和 `recommended_action` 由 `FallRiskPipeline` 融合生成。

## 8. 模块边界

近跌倒事件检测模块不负责以下工作：

- 不做人检。
- 不做人体跟踪。
- 不做姿态估计。
- 不做关键点清洗和平滑。
- 不输出最终 `risk_level`。
- 不输出最终 `recommended_action`。
- 不实现消息推送、电话通知、工单流转或前端看板。
- 不实现个体化长期基线。
- 不把近跌倒和真正跌倒混为一个模块。
- 不输出 `fall_event_score` 或 `long_static_score`。

模块应该只消费 cleaned/smoothed pose JSONL，输出近跌倒事件级 JSONL 和可被融合层消费的 `near_fall_event_score`。

## 12. 风险与取舍

### 主要风险

| 风险 | 影响 |
|---|---|
| 摄像头视角差异 | 正面、侧面、斜侧面和俯视会改变横向偏移、下沉和躯干角的可观测性 |
| 遮挡 | 肩、髋、膝、踝或腕部缺失会导致速度、角度和支撑 proxy 不稳定 |
| 下肢缺失 | 急停、异常下蹲和正常坐下之间更难区分 |
| 腕部缺失或误检 | 支撑 proxy 容易漏判或误判 |
| 跟踪 ID 切换 | 会制造身体中心突变，容易被误判为失衡 |
| 家具不可见 | 无法确认腕部是否真实接触墙、扶手、桌面或床沿 |
| 2D 坐标尺度 | 归一化图像坐标不能直接代表真实速度、距离、高度或三维角度 |
| 正常动作混淆 | 正常坐下、转身、避障、捡物和挥手可能触发近跌倒 proxy |

### 第一版选择规则 baseline 的原因

当前阶段选择规则/统计 baseline，而不是直接依赖 One-Class、Isolation Forest 或时序深度模型，原因是：

- 近跌倒样本少，监督训练不稳定。
- 真实老人场景标注不足，复杂模型容易过拟合模拟数据。
- 规则 baseline 更容易做单元测试和误差复核。
- 输出证据更透明，便于比赛材料解释。
- 规则 baseline 可作为后续异常检测模型的对照组。
- 实时演示路径更稳定，依赖更少。

### 后续升级方向

当有足够正常活动和近跌倒标注后，可升级为：

- One-Class SVM 或 Isolation Forest：使用正常姿态序列学习异常分布。
- 轻量统计模型或 Logistic Regression：校准规则组件权重。
- TCN/GRU/Transformer AutoEncoder：利用更长时序上下文和重构误差识别异常片段。
- 个体化阈值：结合个人历史步态、坐站和活动节律基线。
- 场景物体检测：提高墙面、床边、扶手、桌面等支撑判断可靠性。

这些升级不作为第一版依赖。
