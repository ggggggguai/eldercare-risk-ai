## 3. 冲奖核心命题

作品不以“检测已经发生的跌倒”为最终创新，而以同一系统覆盖三个时间尺度为核心：

| 时间尺度 | 任务 | 输出 | 参考终点 | 可支持的结论与主要指标 |
|---|---|---|---|---|
| 单次功能评估 | 功能风险分层 | 步态、坐站和功能风险分 | TUG、BBS、POMA、SPPB、跌倒史或专家风险分层 | 只能证明与功能或风险 proxy 的关联；报告 AUROC、PR-AUC、Brier 和分层一致性 |
| 天级到周级 | 个人状态变化 | 活动节律和基线偏离 | 重复功能测量、专家确认的状态变化日期或前瞻随访结局 | 证明纵向变化和提前发现；报告变化检测指标、提前发现天数和固定误报下召回 |
| 秒级 | 近跌倒和失衡恢复 | `near_fall_event_score`、事件类型和证据窗 | 人工复核的踉跄、下沉恢复、急停恢复、扶物恢复事件 | 报告事件级 Precision/Recall/F1、每小时误报、onset detection latency 和边界 IoU |
| 事件后 | 疑似跌倒和倒地静止 | 4 级紧急事件 | 官方或人工复核的跌倒、倒地和恢复边界 | 召回率、误报率、检测延迟和恢复状态准确率 |

各类结果必须分别评估。横断面的 TUG、BBS、POMA、SPPB 或跌倒史不能直接支持“提前发现天数”；只有重复测量的状态变化日期或前瞻结局才能支持纵向提前量。公开跌倒视频的事件指标不能作为长期风险预测有效性的证据，临床或功能 proxy 也不能作为精确跌倒事件时间窗的真值。

答辩主叙事固定为：

> 通过天级功能变化、秒级近跌倒和跌倒后紧急响应的多时间尺度建模，结合个体基线、质量感知拒识与萤石服务闭环，将传统事后跌倒识别推进为可验证的前置风险防控。

## 4. 夺奖验收门槛

以下门槛全部满足后，项目才进入“冲击高等级奖项”的材料固化阶段。

### G0：资格与合规门槛

- 确认报名审核已经通过，盖章报名表与系统信息一致，并保留报名凭证。
- 完成真人数据知情同意、脱敏编号、访问控制、保留周期和退出删除流程。
- 建立公开数据、代码、模型权重和图片素材的许可证清单。
- 不让老人模拟跌倒或危险踉跄；危险动作只由健康成年人按安全方案完成。
- 日志、报告和演示不暴露直播地址、令牌、个人身份或未经授权的原始视频。

### G1：数据真值门槛

- 建立统一视频 manifest，记录数据集、文件 hash、真实 FPS、时长、人员、场景、机位、许可和复核状态。
- 所有进入训练或测试的标签均为 `reviewed` 或 `final`，`pending` 和 `uncertain` 不进入最终指标。
- 建立 `risk_labels.jsonl`、`subject_profiles.json` 和 `annotation_review_log.jsonl`。
- 训练、验证、测试按人员、家庭或原视频组划分，同一事件的相邻窗口不得跨集合。
- 测试集在首次正式评估前冻结，调参人员不得查看测试标签。

### G2：算法正确性门槛

- 所有姿态、人体框、速度和下降阈值使用明确且一致的坐标系。
- 标注时间由逐视频真实 FPS 或时间戳换算，并通过帧号与视频时长校验。
- 低质量数据输出拒识、不可用或人工复核状态，不得静默填成正常分数。
- 跌倒状态具有 `suspected`、`confirmed_static`、`recovered` 和 `unresolved` 等算法可观测转换；误报只能由业务或人工复核确认。
- `risk_score`、`risk_level`、`confidence` 和强事件触发的语义一致且可解释。
- 与后端固定版本化事件生命周期契约，明确关联事件 ID、状态更新、schema 版本和误报反馈入口。

### G3：实测指标门槛

- 有可执行的评估命令和固定配置，而不是只在文档中列指标名称。
- 报告事件级 Precision、Recall、F1、PR-AUC、高风险召回率、漏报率和每摄像机小时误报数。
- 报告提前预警时间、事件后检测延迟、P50/P95 推理延迟和端到端回调延迟。
- 报告概率校准指标 Brier score、ECE 或 reliability diagram，以及低质量拒识覆盖率。
- 按人员或家庭 bootstrap 报告 95% 置信区间，并提供原始预测 JSONL。
- 在数据冻结前预注册每个主要结论的最小独立人员、事件或摄像机小时数量，以及可接受的 95% CI 最大宽度；样本不足的结果只能标为探索性结果。

### G4：创新证据门槛

- 分别完成事件、功能评估、纵向基线和配对融合证据族的预注册对照；没有配对数据时不强行制作总融合消融。
- 在相同数据划分和固定误报水平下证明近跌倒或个人基线带来稳定增益。
- 若学习模型没有稳定增益，保留规则主线并如实报告负结果；至少一个核心机制必须在重复实验中产生可复现增益。
- 失败案例中明确说明何时拒识、何时误报以及哪些场景尚不支持。

### G5：萤石与实时工程门槛

- 使用真实萤石设备或开放平台直播地址完成会话启动、推理、事件回调和停止。
- 断流、地址刷新和重连后不会继承旧轨迹、旧窗口或旧跌倒状态。
- 推理与 HTTP 回调解耦，回调失败可重试、可观测且不阻塞视频推理。
- 在固定比赛硬件上完成连续运行、弱网、回调 5xx、多人与遮挡测试。
- 演示前能够输出模型、配置、代码版本和数据版本 hash。

### G6：作品材料门槛

- 专项研究报告中的每个结果都能追溯到报告、配置、数据划分和代码版本。
- 真实直播和录像备份走同一算法服务链路，录像不得冒充实时直播。
- 项目总团队接通算法事件到家属或社区确认、处置记录的最小闭环。
- 源代码、可执行程序、部署说明、功能测试、性能测试、失败案例和合规材料齐全。

## 5. 工作流 A：数据、标注与评估底座

### 5.1 统一数据治理

第一优先级不是增加样本文件，而是让现有数据成为可追溯真值：

1. 为每个视频读取真实 FPS、帧数、时长和文件 hash。
2. 对 CVAT 导出按视频元数据换算时间，不再使用一次性的全局固定 FPS。
3. 恢复或建立 `subject_id`；无法确认人员时，至少建立不会跨集合的 `source_group_id`。
4. 对动作、事件和风险三级标签执行双人复核和冲突仲裁。
5. 将看不清、边界不确定、源文件缺失和标签冲突样本隔离，不进入主指标。
6. 固定版本化 split manifest，所有实验只引用 split ID。
7. 报告标签类别一致性、事件边界偏差和仲裁比例，不能只报告最终标签数量。

各数据资产的角色必须保持清晰：

| 数据 | 主要用途 | 不能证明的结论 |
|---|---|---|
| LE2I/IMViA | 跌倒窗口、居家样例、检测和姿态链路 | 长期个人风险预测 |
| Fall Detection 2017 | fall/ADL 训练、分组测试和困难负样本 | 近跌倒和个人基线 |
| UR Fall | 外部跌倒测试和可选视频/IMU对照 | 单独证明真实居家泛化 |
| Pre_VFallp | 在完成来源、许可和标签语义审计后研究跌倒前症状 | 未审计前不能进入正式结论 |
| TOAGA | 老人步态视频和功能 proxy | 跌倒事件检测 |
| GSTRIDE | 步态、虚弱、TUG/SPPB/FES-I 风险 proxy | 视频姿态链路 |
| LTMM | 长期活动和个体基线 | RGB 跌倒事件 |
| 安全自采数据 | 近跌倒、坐站、困难负样本和居家场景 | 未经专家复核的临床有效性 |

### 5.2 必补样本

近跌倒正样本至少覆盖：

- 横向踉跄后恢复。
- 身体快速下沉后恢复。
- 行走急停后恢复。
- 异常下蹲后恢复。
- 快速扶墙、扶椅或扶手后恢复。
- 失衡但最终未形成稳定倒地。

困难负样本至少覆盖：

- 弯腰捡物、系鞋带和整理床铺。
- 快速坐下、正常坐站和跪地取物。
- 正常转身、绕障、伸展和居家锻炼。
- 家属或护理员进入、交叉和遮挡。
- 助行器、拐杖、宽松衣物和下肢出画。
- 夜间低光、背光、画面压缩和相机抖动。
- 客厅、卧室床边、卫生间门口和楼梯区域。

### 5.3 指标定义

事件级匹配规则必须先固定再评估：

```text
预测事件与真值事件在同一视频、同一任务类型下，
满足时间窗 IoU 阈值或预先定义的 onset 容忍窗口时视为匹配。
一个预测最多匹配一个真值，一个真值最多匹配一个预测。
```

正式评估前必须预注册：

- 每类事件的观察单位、时间窗 IoU、onset 容忍范围和预警搜索窗口。
- 事件合并、去重、恢复和复位规则，以及跨窗口重复告警如何计数。
- 搜索窗口外或无法匹配真值的早期告警如何计为 FP。
- PR-AUC 使用事件候选、固定时间网格还是固定长度窗口作为评价单位；不同单位不得混报。
- 恢复状态准确率的分母、允许恢复时间窗和错误恢复的计数方式。
- 推理延迟从源 PTS 到事件生成，回调延迟从源 PTS 到后端成功确认，两个时钟分别报告。

只有存在后续参考跌倒事件时，才计算正向的跌倒前提前量：

```text
lead_time = reference_event_start - first_level_3_or_higher_alert
```

正值表示提前，负值表示事件发生后才报警。独立近跌倒事件通常报告相对近跌倒 onset 的 `onset_detection_latency`；没有后续跌倒真值时，不把该延迟称为 pre-fall lead time。事件后跌倒检测另行报告 `detection_latency`，不能混入提前量均值。

连续监控必须同时报告：

- `Precision = TP / (TP + FP)`。
- `Recall = TP / (TP + FN)`。
- `F1` 和 `PR-AUC`。
- 每摄像机小时误报数和每家庭日误报数。
- 传统 FPR 与假发现率 `FP / (TP + FP)`，不得都写成“误报率”。
- 事件边界 IoU、恢复召回率、错误恢复率和人工复核工作量。

在首次正式模型比较前冻结统计分析计划，记录主指标、主比较、阈值选择规则、预设分层、缺失数据处理、按人员或家庭聚类的 bootstrap 方法、标注一致性和多模型选择规则。未预注册的新增分析必须标为探索性分析。

## 6. 工作流 B：算法增强与实验矩阵

### 6.1 上游感知与质量控制

- 固定 YOLOv8-Pose + ByteTrack 为可复现工程 baseline。
- 比较 YOLOv8-Pose 与 RTMPose-S/M；只有域内关键点质量和延迟有实际收益时才替换实时主线。
- 对 EMA、One Euro 和 Kalman 进行同输入对照，评价关键点误差、事件峰值保留和附加延迟。
- 用身体尺度、人体框尺度或场景标定统一速度和位移特征，减少摄像机距离影响。
- 保存 `confidence`、`valid mask`、插值来源、异常跳变和 ID 切换信息，不让缺失等价于正常。
- 加入家具、扶手、床边和区域 ROI 证据后，才把腕部动作从 `support_contact_proxy` 升级为接触证据。

### 6.2 统一时序样本

建立可复用的 `[T, V, C]` 或特征时序样本：

```text
joint         = 归一化关键点 + confidence + valid mask
joint_motion  = 一阶和二阶时间差分
bone          = 骨向量、骨长和关节角 proxy
bone_motion   = 骨向量时间差分
metadata      = 真实时间、场景、轨迹质量、插值率和模型版本
```

所有样本必须按真实时间重采样。同一缓存供规则、TCN、ST-GCN++ 和错误分析使用，避免各模型使用不同预处理造成不公平对照。

### 6.3 分支增强路线

| 分支 | 保留 baseline | 第一增强实验 | 数据充足后的增强 | 进入实时主线条件 |
|---|---|---|---|---|
| 步态 | 可解释人工特征规则 | EBM/LightGBM、轻量 TCN | ST-GCN++/CTR-GCN | 独立测试增益、已校准、P95 延迟达标 |
| 坐站 | 状态机和规则评分 | 事件 TCN | MS-TCN++ 相位分割 | 事件边界和局部风险均优于规则 |
| 近跌倒 | 高召回规则候选 | ECOD/IF/OC-SVM 或 hard-negative 分类器 | TCN/ST-GCN++ | 固定召回下显著降低每小时误报 |
| 个体基线 | 分位数、均值和标准差 | Median/MAD、EWMA/CUSUM | PELT、混合效应模型 | 有连续多日独立验证和冷启动策略 |
| 风险融合 | 规则安全覆盖 | Logistic、EBM、LightGBM | 缺失感知时序融合 | 校准、拒识、消融和延迟全部达标 |

### 6.4 个体基线闭环

实时和离线链路统一按日或小时持久化：

- 步态速度、变异、不对称、停顿和转身 proxy。
- 坐站耗时、失败次数、支撑和站稳时间。
- 近跌倒频率、类型和质量。
- 活动量、夜间活动和场景分布。
- 上游模型版本、数据质量和缺失比例。

基线策略必须包含：

- 无历史、初始基线和稳定基线三个阶段。
- 冷启动阶段风险等级和置信度上限。
- 高风险事件、低质量窗口和相机移动不写回正常基线。
- 昼夜、工作日/周末和场景条件基线。
- 上游模型升级后的版本隔离或重新校准。

### 6.5 融合与校准

最终学习模型不能只消费四个风险分数，还应消费局部分支特征、质量、缺失 mask、场景和基线支持度。紧急规则继续保留为安全覆盖，但必须与最终 `risk_score` 和 `confidence` 保持语义一致。

每个融合样本必须记录模态来源、人员、时间范围、采样频率、上游版本和缺失状态。不同数据集中的不同受试者不能拼接成一个“多模态样本”；只有同一人员、同一观察时段具备配对输入和独立风险标签时，才评估最终多模态融合。场景、活动节律或其他信号若都由同一路 RGB 派生，应表述为多分支视觉特征，而不是独立模态。

统一比较：

1. 固定规则评分卡。
2. Logistic Regression。
3. EBM 或 GAM。
4. LightGBM。

对候选模型统一比较 sigmoid、isotonic 等校准方案。测试集只用于一次冻结评估，不参与模型、阈值或校准器选择。

## 7. 工作流 C：实时工程与现场演示

### 7.1 推理链路

- 拉流线程和推理线程解耦，使用容量 1-2 的 latest-frame 队列，防止处理旧帧。
- 同时记录源 PTS、接收时间、推理完成时间和事件回调时间。
- 对姿态清洗和窗口特征做增量或缓存计算，不每 0.5 秒全量重算 10 秒窗口。
- 每次断流重连或更新直播地址后创建新 `stream_epoch`，原子重置 tracker、特征窗口和状态机。
- 多人画面增加 ROI、初始目标确认和身份冲突状态；无法确认老人身份时拒绝输出个人风险。

### 7.2 事件与回调

- 将推理线程输出写入有界、持久化 outbox。
- 投递线程复用稳定 `event_id` 重试，只有确认成功后才更新已投递状态。
- 近跌倒、疑似跌倒和倒地静止绕过普通融合节流。
- 算法只输出可观测的 `suspected`、`confirmed_static`、`recovered` 或 `unresolved` 状态；`confirmed_false_alarm` 由业务或人工复核反馈。
- 与后端共同版本化生命周期契约，明确 `event_id`、关联事件 ID、事件状态、schema 版本、恢复回调和误报反馈入口。
- `risk_factors` 保持稳定机器码，中文解释由展示层映射，不写入算法判定逻辑。
- 停止接口只有在线程和资源实际退出后才报告 `stopped`。

### 7.3 可观测性与 SLO

状态接口和日志至少暴露：

- 当前流 epoch、帧龄、实际处理 FPS 和丢帧数。
- 主目标 ID、目标缺失时间和多人冲突状态。
- 姿态质量、最后分析时间和低质量拒识原因。
- 推理 P50/P95、最后事件时间和端到端回调延迟。
- 重连次数、outbox 深度、回调成功率和最后失败原因。
- 模型、配置、代码和数据版本 hash。

## 8. 工作流 D：萤石平台与应用闭环

算法仓库继续只负责算法服务和 `module=fall_risk` 事件，不实现 App、消息通道、账号、工单或线下处置。但是项目总团队要完成比赛要求的最小应用闭环：

```text
  -> 业务后端持久化
  -> 家属或社区确认
  -> 处置记录和事件闭环
```

必须保留以下比赛证据：

- 萤石应用、设备和能力调用的脱敏配置说明。
- 真实会话启动、地址更新、断流重连和停止日志。
- 至少一个 3 级和一个 4 级事件的完整回调及业务确认记录。
- 客厅、卧室或卫生间至少三类典型居家场景测试。
- 真实直播主演示和实时速率录像备份，两者明确标识数据源。
- 固定比赛硬件、网络条件、FPS、延迟、CPU/GPU、内存和连续运行时长。

## 11. 实验与消融设计

### 11.1 分任务对照组

不同数据集并非同一批人员的配对模态，不能把公开跌倒视频、功能评估和长期活动数据串成一个逐级 A-F 总消融。实验拆成四个相互独立的证据族：

| 实验证据族 | 主要对照 | 必需数据 | 可支持的结论 |
|---|---|---|---|
| 跌倒与近跌倒事件 | 事后跌倒检测；规则近跌倒；规则候选加分类器；去质量门控；去恢复约束 | 同一批具有事件边界、困难负样本和质量标签的视频 | 秒级事件识别、误报、onset 延迟和恢复约束价值 |
| 步态与坐站功能 | 规则；EBM/LightGBM；轻量 TCN；坐站状态机与事件 TCN | 同一批具有人员级功能 proxy、步态或坐站标签的样本 | 功能风险分层和局部分支建模价值 |
| 个体基线纵向 | 群体阈值；滚动均值；Median/MAD；EWMA/CUSUM；去污染保护 | 同一人员连续多日观测和重复功能或专家状态变化标签 | 个体化相对群体阈值的稳定性和提前发现价值 |
| 配对风险融合 | 规则；Logistic；EBM；LightGBM；分别去场景、去活动节律、去质量拒识和去个人基线 | 同一人员同一时段具备全部输入、来源记录和独立风险标签 | 多来源融合、校准和缺失处理的增益 |

每个证据族内部必须使用相同数据划分、相同上游缓存和相同评价协议。只改变一个因素，不能同时替换检测、姿态、平滑和融合后把结果归因于某一模块。若场景、活动节律和姿态全部由同一路 RGB 派生，报告中称为多分支视觉特征；只有来源独立、人员与时间对齐的输入才称为多模态融合。

### 11.2 鲁棒性矩阵

每个最终候选至少按以下维度分层：

| 维度 | 分层 |
|---|---|
| 场景 | 客厅、卧室床边、卫生间门口、楼梯或普通 home |
| 光照 | 白天、夜间、背光、低照度 |
| 视角 | 正面、侧面、斜侧、高位安装 |
| 可见性 | 清晰、局部遮挡、下肢出画、短时完全遮挡 |
| 人数 | 单人、护理员进入、多人交叉 |
| 行动方式 | 正常行走、慢走、拖步、坐站、转身、助行器 |
| 网络 | 正常、短时断流、地址刷新、回调延迟和 5xx |

### 11.3 内部目标值

下表是冲奖目标，不是当前实测结果：

| 指标 | 内部目标 | 说明 |
|---|---:|---|
| 高风险事件 Recall | `>= 90%` | 观察单位为冻结事件集中的参考 3-4 级事件；同时报告 95% CI 和漏报案例 |
| 近跌倒事件 F1 | `>= 80%` | 观察单位为独立、已复核且一对一匹配的近跌倒事件 |
| 普通风险分类 Accuracy | `>= 85%` | 仅在具备独立 `risk_labels` 的预注册窗口或日级记录上计算；不能替代 PR-AUC |
| 单路算法推理延迟 | `P95 < 1 s` | 固定比赛硬件，从源帧 PTS 到事件生成 |
| 算法事件回调延迟 | `P95 < 3 s` | 从源帧 PTS 到业务后端成功确认 |
| 近跌倒输出延迟 | `P95 <= 1 s` | 从近跌倒证据满足到事件生成，越低越好 |
| 连续误报评估规模 | `>= 24` 摄像机小时 | 至少覆盖 3 类场景；每小时误报上线在首轮 baseline 后预注册 |
| 主指标统计精度 | 95% CI 半宽目标 `<= 10` 个百分点 | 未满足时只作为探索性结果，不作稳定泛化结论 |
| 连续运行稳定性 | 单次长时烟测 `>= 8 h` | 报告断流恢复、回调成功率和资源峰值 |

任何未达到目标的指标必须原样报告，并通过失败案例、场景边界和改进计划解释，不能只展示表现最好的子集。

| 平台与业务 | 萤石权限、直播地址、后端回调和处置闭环 | 联调记录和端到端验收 |
| 合规与研究 | 知情同意、许可证、专项报告和引用 | 合规清单和报告章节 |
| 演示与答辩 | 演示脚本、可视证据、问题库和材料一致性 | 周演练录像和问题闭环 |

同一人可以兼任多个角色，但数据划分、模型调参和最终盲测应尽量由不同成员执行。

## 14. 风险登记与决策规则

| 风险 | 预警信号 | 应对策略 | 停止扩展条件 |
|---|---|---|---|
| 近跌倒样本不足 | 正例类型单一、CI 很宽 | 优先安全采集和 hard negatives，先做规则加单类模型 | 没有独立正例时不上有监督大模型 |
| 人员身份无法恢复 | `subject_id` 长期 unknown | 使用严格 source group 外部测试并披露限制 | 不做按人风险泛化声明 |
| 萤石权限延迟 | 第 1 周仍无真实调用 | 立即联系发榜单位，保留同接口本地演示 | 不把本地文件称为萤石联调 |
| 指标未达目标 | 验证集误报高或跨域下降明显 | 收敛场景边界、强化拒识、保留规则主线 | 不为了数字反复查看测试集 |
| 复杂模型拖慢实时 | P95 超预算或依赖不稳定 | 只留离线对照，实时回退轻量模型 | 未证明增益前不上主演示 |
| 现场网络不稳定 | 断流或回调失败频繁 | 真实直播主路径加同链路录像备份 | 备份必须明确标记，不冒充直播 |
| 合规或许可不清 | 数据来源、权重许可无法确认 | 隔离相关资产并更换可用来源 | 未完成审计的数据不进入提交物 |
| 范围失控 | 同时扩展多个风险方向或多个大模型 | 聚焦跌倒方向和官方高权重证据 | P0 门槛未过前冻结 P2 探索 |

模型升级必须遵循以下决策规则：

1. 新模型先离线复现，再进入相同 split 的公平对照。
2. 只有独立验证集指标、校准、延迟和稳定性同时达标时，才进入实时候选。
3. 每轮只替换一个模块，并保留旧模型一键回退。
4. 测试集、报告数字和失败案例不得因模型表现不佳而删除或重划。
5. Transformer、RGB 视频大模型和单目 3D 只作为有明确假设的增强实验，不作为“模型数量”展示。

## 15. 周度治理和完成判定

每周固定进行一次证据审查，只回答以下问题：

1. 本周新增了什么可执行代码、数据或报告？
2. 哪个指标发生变化，变化是否来自相同数据和配置？
3. 新模型是否真正优于 baseline，还是只增加复杂度？
4. 当前最大漏报和误报场景是什么？
5. 萤石真实链路和业务闭环是否可重复运行？
6. 是否出现隐私、许可、标签泄漏或材料不一致风险？

状态维护规则：

- 本计划只维护目标、阶段和验收门槛。
- 尚未完成的实际任务统一写入[当前任务](../../../tasks/README.md)。
- 完成能力更新[模块 README](../README.md)和[算法工程架构](../../../architecture/算法工程骨架.md)。
- 新实测结果写入 `reports/`，不得回填成本文中的“已实现”。
- 已结束的计划阶段或比赛版本整体移入 `docs/archive/`。

最终完成判定不是“代码全部写完”，而是：

```text
数据可追溯
+ 指标可复现
+ 创新增益可消融
+ 萤石闭环可演示
+ 故障场景可恢复
+ 合规材料无缺项
+ 报告与实际输出一致
= 具备冲击高等级奖项的提交条件
```
