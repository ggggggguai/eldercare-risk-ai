# 基于多模态 AI 的居家老人跌倒风险算法研发计划

## 1. 项目定位

本项目聚焦比赛方案中的“跌倒风险”方向，目标不是做传统的“跌倒动作检测”，而是构建面向居家养老场景的跌倒风险前置预警算法系统。

核心目标：

- 从“跌倒后识别”升级为“跌倒前风险预测、失衡中预警、跌倒后响应”。
- 基于视频、姿态、行为节律、场景环境等多模态信息，输出老人跌倒风险等级。
- 建立个体化行为基线，识别老人相对于自身历史状态的风险变化。
- 形成可运行算法原型、测试报告和专项研究报告。

最终系统输出：

```text
输入：萤石设备或开放平台提供的居家视频流、场景信息、历史行为数据
输出：跌倒风险等级、风险分数、置信度、风险原因、预警动作建议
```

### 1.1 算法模块边界

本文件只覆盖总项目中的“跌倒风险算法模块”。心理健康风险、诈骗识别、家属端、社区端、消息推送、平台账号体系和告警工单流转由其他模块或开发部门承接。

本算法模块负责：

- 从视频流和结构化场景信息中识别跌倒风险。
- 输出跌倒风险等级、风险分数、置信度和风险因子。
- 检测近跌倒、起身困难、转身不稳、长时间倒地等风险事件。
- 建立个人行为基线，判断步态、坐站、夜间活动等指标是否异常偏离。
- 给出标准化预警动作建议，供平台或业务系统调用。

本算法模块不负责：

- 家属 App、社区端后台或平台页面开发。
- 消息推送通道、电话通知、工单流转和账号权限系统。
- 养老服务人员调度或线下处置流程。

### 1.2 对外接口与萤石平台适配

算法输入优先适配萤石摄像头或萤石开放平台的视频流、设备编号、时间戳和基础场景元数据。若比赛演示阶段无法直接获取实时平台流，则保留离线视频推理入口，但接口字段保持一致，便于后续接入萤石开放平台。

对外输出采用事件化 JSON，平台开发只需消费算法结果，不需要理解模型内部细节：

```json
{
  "module": "fall_risk",
  "device_id": "ezviz_cam_001",
  "person_id": "elder_001",
  "timestamp": "2026-06-26T14:20:00+08:00",
  "scene_region": "bedroom_bedside",
  "risk_level": 3,
  "risk_score": 0.84,
  "confidence": 0.87,
  "trigger_event": "near_fall",
  "risk_factors": [
    "重心快速横向偏移",
    "步态节律中断",
    "手部快速接触墙面"
  ],
  "recommended_action": "notify_guardian",
  "evidence_window": {
    "start_time": 15.2,
    "end_time": 17.6
  },
  "model_version": "fall-risk-v1.0"
}
```

`recommended_action` 只表示算法建议，不直接实现消息推送。建议动作编码：

| 编码                | 含义     | 算法触发条件             |
| ----------------- | ------ | ------------------ |
| `record_only`     | 仅记录    | 0 级正常或低置信度轻微异常     |
| `observe`         | 持续观察   | 1 级低风险或轻微偏离个人基线    |
| `remind_user`     | 建议提醒老人 | 2 级中风险、起身困难或活动节律异常 |
| `notify_guardian` | 建议通知家属 | 3 级高风险、近跌倒或连续异常    |
| `emergency_alert` | 建议紧急告警 | 4 级疑似跌倒或长时间倒地      |
| `manual_review`   | 建议人工复核 | 模型置信度不足、遮挡严重或多事件冲突 |

风险等级：

```text
0级：正常
1级：低风险，持续观察
2级：中风险，提醒老人或家属关注
3级：高风险，主动推送预警
4级：紧急风险，疑似跌倒或长时间倒地，立即告警
```

## 2. 技术路线总览

整体采用“可落地主线 + 增强模型实验”的分层路线。比赛演示和实测验证优先保证稳定、可解释、低延迟；复杂深度模型作为增强实验和消融对比，不作为唯一可运行路径。

主算法路线：

```text
萤石设备或开放平台视频流
  ↓
人体检测与跟踪
  ↓
人体姿态关键点提取
  ↓
关键点质量控制与时序平滑
  ↓
步态稳定性分析
  ↓
坐站转换能力分析
  ↓
近跌倒事件检测
  ↓
个体化行为基线建模
  ↓
轻量风险融合模型 + 规则校准
  ↓
跌倒风险等级 + 置信度 + 可解释风险因子 + 预警动作建议
```

核心模型组合按“主线交付优先、增强实验备选”的原则组织：

1. **人体检测与跟踪**

   - 主线方法：萤石视频流 + YOLOv8/人体检测 + ByteTrack

   - 增强实验：更高精度检测器

   - 作用：定位老人并保持身份连续

2. **姿态估计**

   - 主线方法：RTMPose

   - 增强实验：其他姿态模型对比

   - 作用：提取人体关键点

3. **姿态质量控制**

   - 主线方法：关键点置信度过滤 + 平滑

   - 增强实验：时序补全模型

   - 作用：降低遮挡、低光造成的抖动

4. **姿态动作识别**

   - 主线方法：可解释动作特征 + 轻量 TCN

   - 增强实验：ST-GCN

   - 作用：识别拖步、转身不稳、起身困难等动作

5. **步态风险建模**

   - 主线方法：规则校准 + 轻量分类/回归模型

   - 增强实验：TCN

   - 作用：输出步态稳定性风险

6. **近跌倒检测**

   - 主线方法：特征突变检测 + One-Class/Isolation Forest

   - 增强实验：Transformer AutoEncoder

   - 作用：检测异常姿态序列

7. **个体基线建模**

   - 主线方法：滚动均值、标准差、分位数偏离

   - 增强实验：个体时序模型

   - 作用：判断个人状态变化

8. **风险融合**

   - 主线方法：规则评分卡 + Logistic/LightGBM

   - 增强实验：Risk Fusion Transformer

   - 作用：输出综合风险等级

主线方法必须在小样本数据下可训练、可解释、可复现。ST-GCN、Transformer AutoEncoder 和 Risk Fusion Transformer 只在数据量、训练稳定性和延迟满足要求后进入增强版，不作为比赛交付的单点依赖。

## 3. 跌倒风险特征体系

### 3.1 步态稳定性特征

重点识别老人行走能力是否下降。

可提取特征：

- 步速
- 步频
- 步幅
- 左右步态对称性
- 躯干左右摆动幅度
- 重心横向偏移
- 行走路径偏移
- 拖步、小碎步
- 行走中急停次数
- 转身耗时和转身角速度

风险判断示例：

```text
步速较个人基线下降 25%
+ 重心摆动幅度增加
+ 转身耗时增加
= 跌倒风险升高
```

### 3.2 姿态控制特征

重点识别老人身体控制能力是否下降。

关键动作：

- 坐下
- 起身
- 弯腰
- 转身
- 夜间下床
- 扶墙或扶家具移动

可提取特征：

- 坐站转换时间
- 起身失败次数
- 起身时躯干前倾角
- 起身后稳定时间
- 弯腰后恢复时间
- 站立时身体摇晃程度
- 手部是否快速接触墙面或家具

### 3.3 近跌倒事件特征

近跌倒事件是前置预警的核心。

需要识别的事件：

- 突然踉跄
- 快速扶墙
- 身体重心突然横向偏移
- 身体快速下沉但未倒地
- 行走中急停并恢复
- 异常下蹲后恢复

这些事件不一定是真跌倒，但对风险预警价值高。

### 3.4 活动节律特征

跌倒风险通常会在长期行为变化中提前暴露。

按天统计：

- 日均活动时长
- 房间转移次数
- 行走总距离
- 久坐或久卧时长
- 夜间起床次数
- 凌晨活动次数
- 卫生间停留时间
- 床边停留时间

趋势指标：

- 近 7 天步速是否持续下降
- 近 7 天夜间活动是否增加
- 近 14 天活动量是否下降
- 是否频繁出现在卫生间、床边等高风险区域

### 3.5 场景环境风险特征

不同居家区域风险不同。

重点场景：

- 客厅
- 卧室
- 卫生间门口
- 厨房
- 走廊
- 床边
- 沙发边

场景风险因素：

- 夜间低照度
- 地面可能湿滑
- 活动区域狭窄
- 障碍物遮挡
- 老人长时间独处
- 高风险区域异常静止

## 4. 算法模块设计

### 4.1 人体检测与跟踪模块

目标：

- 从视频中检测老人位置。
- 保持同一老人跨帧身份一致。
- 输出人体框、轨迹、速度和所在区域。

输入：

```text
视频帧序列
```

输出：

```json
{
  "frame_id": 120,
  "person_id": "elder_001",
  "bbox": [x1, y1, x2, y2],
  "scene_region": "living_room",
  "track_confidence": 0.94
}
```

确定方法：

- 人体检测：YOLOv8
- 跟踪关联：ByteTrack

### 4.2 姿态关键点模块

目标：

- 提取人体骨架关键点。
- 将原始视频转化为结构化姿态序列。
- 为后续时序动作识别和风险建模提供结构化输入。

关键点：

```text
头部、肩部、肘部、手腕、髋部、膝盖、脚踝
```

输出：

```json
{
  "frame_id": 120,
  "person_id": "elder_001",
  "keypoints": [
    {"name": "left_hip", "x": 0.42, "y": 0.61, "score": 0.91},
    {"name": "right_hip", "x": 0.50, "y": 0.62, "score": 0.89}
  ]
}
```

确定模型：

- RTMPose：负责人体骨架关键点提取，输出肩、髋、膝、踝等关键点序列。

### 4.3 姿态时序动作识别模块

目标：

- 识别风险相关动作。
- 不只判断是否跌倒，而是识别风险动作模式。

动作类别：

```text
正常行走
缓慢行走
拖步
小碎步
转身不稳
坐下
正常起身
起身困难
弯腰恢复慢
扶墙行走
踉跄恢复
模拟跌倒
长时间倒地
```

主线方法：

- 从关键点序列中提取可解释时序特征，如步速、躯干倾角、重心偏移、起身耗时和转身稳定性。
- 使用轻量 TCN 或传统分类器识别风险动作，保证小样本下可训练、可解释。
- ST-GCN 作为增强实验，用于与主线方法对比，不作为比赛演示的单点依赖。

输出：

```json
{
  "action": "unstable_turning",
  "confidence": 0.86,
  "start_time": 8.4,
  "end_time": 11.2
}
```

### 4.4 步态稳定性评分模块

目标：

- 评估老人行走是否稳定。
- 输出连续的步态风险分数。

核心指标：

```text
步速
步频
步幅
左右对称性
重心摆动
躯干倾角
路径偏移
转身稳定性
```

主线模型：

```text
关键点序列 → 步态特征提取 → 规则校准/轻量回归模型 → GaitRiskScore
```

输出：

```json
{
  "gait_risk_score": 0.72,
  "risk_factors": [
    "步速下降",
    "重心横向摆动增大",
    "转身耗时增加"
  ]
}
```

### 4.5 坐站转换风险模块

目标：

- 识别老人从坐姿到站姿、从站姿到坐姿过程中的能力下降。
- 该模块适合作为比赛演示亮点。

关键指标：

- 起身耗时
- 起身失败次数
- 起身过程中躯干前倾角
- 起身后摇晃程度
- 是否借助扶手、桌面、墙面
- 站稳所需时间

输出：

```json
{
  "sit_stand_risk_score": 0.66,
  "duration": 4.8,
  "failed_attempts": 1,
  "risk_factors": [
    "起身耗时较长",
    "起身后存在明显摇晃"
  ]
}
```

### 4.6 近跌倒异常检测模块

目标：

- 检测未真正摔倒但存在明显风险的异常姿态序列。
- 支撑“前置预警”创新点。

主线方法：

```text
关键点时序特征 → 突变检测 + One-Class/Isolation Forest → NearFallEventScore
```

增强实验：

```text
正常动作序列 → Transformer AutoEncoder → 重构误差
```

判断逻辑：

```text
异常分数显著升高
+ 姿态速度突变
+ 重心偏移异常
= 疑似近跌倒事件
```

输出：

```json
{
  "event_type": "near_fall",
  "confidence": 0.83,
  "start_time": 15.2,
  "end_time": 17.6,
  "evidence": [
    "重心快速横向偏移",
    "手部快速接触墙面",
    "步态节奏中断"
  ]
}
```

### 4.7 个体化行为基线模块

目标：

- 建立每位老人的正常行为模式。
- 判断当前状态是否偏离个人历史水平。

基线周期：

```text
初始基线：3-7 天
稳定基线：7-14 天
```

基线特征：

- 平均步速
- 平均坐站耗时
- 夜间活动次数
- 日均活动量
- 常见活动区域
- 转身稳定性
- 近跌倒事件频率

风险偏离计算：

```text
当前指标 - 个人历史均值
------------------------
个人历史标准差
```

输出：

```json
{
  "baseline_deviation_score": 0.78,
  "deviation_factors": [
    "步速较个人基线下降 24%",
    "夜间起床次数增加 80%",
    "坐站转换时间增加 35%"
  ]
}
```

### 4.8 多模态风险融合模块

目标：

- 融合不同风险来源。
- 输出最终跌倒风险等级。

输入特征：

```text
GaitRiskScore
SitStandRiskScore
NearFallEventScore
BaselineDeviationScore
SceneRiskScore
ActivityRhythmScore
```

主线模型：

```text
规则评分卡 + Logistic/LightGBM
```

增强实验：

```text
Risk Fusion Transformer
```

输出：

```json
{
  "risk_level": 3,
  "risk_score": 0.84,
  "confidence": 0.87,
  "risk_factors": [
    "检测到近跌倒事件",
    "步态稳定性下降",
    "夜间床边活动频繁"
  ],
  "recommended_action": "notify_guardian"
}
```

## 5. 工程原型设计

### 5.1 模块划分

```text
pose/
  人体检测、跟踪、关键点提取

features/
  步态、坐站、转身、活动节律特征

models/
  主线轻量模型、可解释评分卡、增强模型实验

inference/
  单视频推理、实时流推理、风险融合、事件化接口输出

dashboard/
  算法演示看板、风险曲线、事件时间轴

reports/
  测试报告、实验结果、提交材料
```

### 5.2 萤石平台适配接口

算法工程只负责输入输出适配，不负责平台业务开发。

输入适配：

```text
设备视频流或离线视频
device_id
timestamp
scene_region
person_id 或临时 track_id
可选环境信息：低照度、摄像头安装位置、区域标签
```

输出适配：

```text
事件化 JSON
风险等级
风险分数
置信度
风险因子
证据时间窗
算法建议动作
模型版本
```

平台或业务系统可根据 `recommended_action` 自行决定是否推送、如何推送、推送给谁。算法模块只保证输出稳定、字段清晰、含义可解释。

### 5.3 推理流程

```text
读取视频
  ↓
检测老人
  ↓
提取关键点
  ↓
识别动作片段
  ↓
计算步态与姿态风险
  ↓
检测近跌倒事件
  ↓
读取个人基线
  ↓
融合风险分数
  ↓
输出风险等级和解释
```

### 5.4 判断频率与事件 JSON 输出策略

当前第一版工程实现中，风险融合模块是单次特征输入触发式推理：每调用一次 `predict_from_features()`，输出一条风险事件 JSON。它还没有实现实时视频流上的自动定时调度。

后续接入视频流、姿态时序动作识别和风险融合后，推理调度采用“逐帧结构化、滑窗判断、事件节流输出”的策略：

```text
视频帧输入 15-25 FPS
  ↓
人体检测、跟踪、姿态关键点逐帧输出结构化 JSONL
  ↓
姿态时序动作识别按 1-2 秒滑窗滚动判断
  ↓
风险融合默认每 2 秒更新一次内部风险状态
  ↓
普通状态每 5 秒输出一次汇总风险 JSON
  ↓
高风险或紧急事件立即输出事件 JSON
```

建议频率：

| 场景            | 判断频率                     | JSON 输出策略               |
| ------------- | ------------------------:| ----------------------- |
| 人体检测、跟踪、姿态关键点 | 按帧处理，演示主路径可降采样到 5-10 FPS | 保存结构化 JSONL，不作为每帧告警     |
| 姿态时序动作识别      | 1-2 秒滑窗滚动判断              | 输出动作片段或中间特征，供风险融合使用     |
| 普通低风险状态       | 每 2 秒更新风险状态              | 每 5 秒输出一次状态型风险 JSON     |
| 步态、坐站、转身风险    | 每 2 秒滚动判断                | 风险等级变化时立即输出，否则按 5 秒节流输出 |
| 近跌倒事件         | 0.5-1 秒内确认               | 立即输出事件型风险 JSON          |
| 疑似跌倒或长时间静止    | 立即判断，必要时持续确认             | 立即输出 4 级紧急风险 JSON       |
| 个体化行为基线       | 按小时或按天聚合                 | 输出趋势型风险因子，不按帧输出         |

输出原则：

- 不对每一帧都输出风险告警 JSON，避免平台侧收到大量重复事件。
- 普通状态采用节流输出，用于前端状态刷新和日志记录。
- 风险等级升高、触发事件变化、建议动作升级时立即输出。
- 高风险和紧急风险不等待 5 秒节流周期，必须及时输出。
- 同一事件在短时间内重复触发时，需要合并证据时间窗，避免重复告警。
- 所有事件 JSON 必须包含 `timestamp` 和 `evidence_window`，便于回放、复核和报告统计。
- 离线视频推理可以按同样策略模拟实时输出，报告中应说明这是离线复现的事件时间线。

## 6. 测试与评价

### 6.1 核心指标

分类指标：

- Accuracy
- Precision
- Recall
- F1-score
- AUC

风险预警指标：

- 高风险召回率
- 误报率
- 近跌倒事件召回率
- 平均提前预警时间
- 风险等级稳定性

系统指标：

- 单视频处理时间
- 实时推理延迟
- 算法事件输出延迟
- 关键点提取成功率

### 6.2 基线方法

为证明算法有效，需要设置至少 3 类基线：

| 基线       | 内容                | 目的             |
| -------- | ----------------- | -------------- |
| 规则阈值基线   | 步速、重心偏移、倒地静止等手工阈值 | 证明学习模型是否真正带来提升 |
| 单一动作识别基线 | 只识别跌倒或异常动作        | 证明风险前置预警优于事后识别 |
| 无个体基线模型  | 不使用个人历史数据         | 证明个体化基线的价值     |

### 6.3 目标指标

建议按事件识别、风险融合和实时性能分别报告：

```text
高风险召回率 >= 90%
近跌倒事件 F1-score >= 80%
普通风险分类 Accuracy >= 85%
必须报告 Accuracy、Precision、Recall、F1-score、误报率、漏报案例

实时性能：
单路视频推理延迟 < 1 秒
算法事件输出延迟 < 3 秒
```

### 6.4 指标定义

提前预警时间：

```text
提前预警时间 = 算法首次输出 3 级及以上风险的时间 - 参考跌倒或近跌倒事件开始时间
```

若是长期风险趋势预警，则单独报告：

```text
长期风险提前发现时间 = 算法首次输出连续 2 级及以上风险的日期 - 参考风险状态变化日期
```

误报率：

```text
误报率 = 被算法判为 3 级及以上但复核为 0-1 级的事件数 / 算法输出 3 级及以上事件总数
```

漏报率：

```text
漏报率 = 参考 3-4 级事件中算法输出低于 3 级的事件数 / 参考 3-4 级事件总数
```

关键点提取成功率：

```text
关键点提取成功率 = 有效关键点帧数 / 总帧数
```

有效关键点帧要求肩、髋、膝、踝等核心点置信度达到阈值；低于阈值的片段进入低置信度处理逻辑。

### 6.5 必测场景

- 正常白天客厅行走，输出 0-1 级风险。
- 拖步、小碎步、摇晃行走，输出 2 级以上风险。
- 起身困难，输出坐站风险因子。
- 转身不稳，输出转身风险因子。
- 踉跄后扶墙恢复，识别近跌倒事件。
- 夜间床边活动频繁，提高风险等级。
- 卫生间门口长时间静止，触发高风险预警。
- 模拟跌倒后倒地不动，触发 4 级紧急告警。
- 摄像头轻微遮挡，系统仍能稳定输出。
- 同一老人连续数日步速下降，系统基于个人基线提高风险等级。

### 6.6 运行性能预算

为了避免复杂模型导致演示不稳定，单路视频按 15-25 FPS 输入进行预算：

| 环节        | 目标耗时     | 备注               |
| --------- | --------:| ---------------- |
| 抽帧与预处理    | < 80 ms  | 可按 5-10 FPS 分析风险 |
| 人体检测与跟踪   | < 250 ms | 可复用平台检测能力时进一步降低  |
| 姿态关键点提取   | < 300 ms | 低置信度帧可跳过或插值      |
| 特征计算      | < 100 ms | 步态、坐站、重心、活动节律    |
| 风险融合与事件输出 | < 100 ms | 主线轻量模型优先         |
| 总推理延迟     | < 1 s    | 超时则降级为轻量规则路径     |

若增强模型超过性能预算，只用于离线报告和消融实验，不进入现场实时演示主路径。

实时事件输出还需满足：

| 输出类型            | 目标延迟             |
| --------------- | ----------------:|
| 普通状态型风险 JSON    | 每 5 秒刷新一次        |
| 风险等级变化事件        | 变化后 1 秒内输出       |
| 近跌倒事件 JSON      | 事件发生后 0.5-1 秒内输出 |
| 疑似跌倒或长时间静止 JSON | 判断成立后立即输出        |

## 7. 消融实验设计

为证明创新增强路线有效，需要做消融实验。

实验组：

```text
完整模型：
姿态时序模型 + 近跌倒检测 + 个体基线 + 多模态融合
```

消融组：

```text
去掉个体化基线
去掉近跌倒检测
去掉活动节律特征
去掉场景风险特征
只使用姿态动作识别
只使用规则阈值
```

预期结论：

- 加入个体化基线后，高风险识别更稳定。
- 加入近跌倒检测后，前置预警能力更强。
- 加入活动节律后，长期风险趋势更明显。
- 完整模型优于单一动作识别模型。

## 9. 主要风险与应对

### 风险 1：近跌倒事件稀少

影响：

- 监督学习困难。

应对：

- 主线使用特征突变检测、One-Class/Isolation Forest 等小样本友好方法。
- Transformer AutoEncoder 作为增强实验，不作为唯一检测路径。
- 将高重构误差、重心突变、手部扶墙、步态中断等证据组合后再输出近跌倒。
- 对近跌倒阈值进行复核验证，单独报告误报来源。

### 风险 2：摄像头角度变化导致关键点不稳定

影响：

- 姿态特征噪声大。

应对：

- 增加多视角输入适配。
- 使用关键点平滑。
- 过滤低置信度关键点。
- 增加遮挡和低光场景测试。

### 风险 3：模型复杂但演示不稳定

影响：

- 终审演示风险高。

应对：

- 将轻量风险融合模型作为实时演示主路径。
- 复杂深度模型只作为增强实验和消融对比。
- 设置推理超时降级策略：关键点质量低或模型耗时超限时，回退到规则评分卡。
- 保留离线视频演示模式，但报告中明确离线和实时结果差异。

### 风险 4：萤石平台适配不足

影响：

- 比赛要求基于萤石开放平台，若算法无法说明输入输出适配，会影响得分。

应对：

- 算法输入字段兼容设备视频流、`device_id`、时间戳和场景区域。
- 算法输出统一事件化 JSON，供平台开发或业务系统消费。
- 实时接入不可用时保留离线视频推理，但接口字段与平台接入保持一致。
