# 第 6 步：坐站转换能力分析执行计划

## 1. 当前状态

### 事实

- 固定算法路线已经推进到“步态稳定性分析 baseline”。
- `tracking.py`、`pose.py`、`pose_quality.py`、`gait.py` 已存在。
- `pose_quality.py` 已输出下游可用的清洗后姿态字段，包括：
  - `keypoints[].valid`
  - `keypoints[].source`
  - `keypoints[].x_smooth`
  - `keypoints[].y_smooth`
  - `keypoints[].is_jump_outlier`
  - `core_keypoint_quality`
  - `window_quality.usable_for_sit_stand`
- `gait.py` 已实现独立的步态稳定性规则 baseline，并按窗口输出 `gait_risk_score`、中间特征、质量覆盖和风险因子。
- `FallRiskPipeline` 已能消费结构化特征字段 `sit_stand_risk_score`，并在最终融合阶段生成 `risk_score`、`risk_level`、`recommended_action`。
- 仓库当前尚未实现独立的坐站转换能力分析模块，尚无 `sit_stand.py`、坐站运行脚本和坐站单元测试。

### 推断

- 坐站转换模块应沿用步态模块的工程形态：独立模块、独立脚本、独立测试、从 cleaned pose JSONL 读取数据，输出结构化 JSONL。
- 坐站模块输出应作为风险融合层的输入，而不是直接替代融合层。
- 第一版应优先服务比赛演示和算法链路闭环，采用可解释、可测试、可回放的规则/统计 baseline。

### 建议

- 不要把坐站逻辑塞进 `pose_quality.py` 或 `pipeline.py`。
- 不要在坐站模块内输出最终 `risk_level` 或 `recommended_action`。
- 坐站模块可以输出局部 `sit_stand_risk_score`、局部风险因子和证据字段；最终是否属于整体跌倒风险，由后续风险融合模块结合步态、近跌倒、个体基线、场景等信号判断。
- 第一版不要依赖深度动作识别模型，也不要假设已有真实老人长期坐站风险标签。

当前代码与文档的主要差距是：文档中已定义坐站风险方向，`pipeline.py` 也已预留 `sit_stand_risk_score`，但坐站特征提取模块本身尚未落地。

## 2. 目标与边界

### 目标

坐站转换能力分析对应固定算法路线第 6 步：

```text
步态稳定性分析
  -> 坐站转换能力分析
  -> 近跌倒事件检测
```

该模块目标是识别老人从坐姿到站姿、从站姿到坐姿过程中的能力下降，重点输出：

- 起身或坐下耗时是否变长。
- 是否存在一次或多次起身失败。
- 起身过程中躯干是否明显前倾。
- 起身后是否明显摇晃。
- 是否疑似借助扶手、桌面、墙面等支撑。
- 起身后站稳所需时间。

### 模块输入

输入为 `pose_quality.py` 输出的 cleaned/smoothed pose JSONL，每行表示一人一帧。

必须优先使用：

- `x_smooth` / `y_smooth`
- `valid`
- `is_jump_outlier`
- `window_quality.usable_for_sit_stand`
- `core_keypoint_quality`

关键点优先级：

- 必需核心点：左右肩、左右髋、左右膝、左右踝。
- 可选增强点：左右腕。

腕部关键点只能作为“疑似支撑使用”的 proxy，不应作为第一版质量门控的必需核心点。

### 模块输出

输出为坐站事件级 JSONL，每条记录表示一次候选坐站转换事件，包含：

- `sit_stand_risk_score`
- 坐站中间特征。
- 质量覆盖。
- 风险因子。
- 模型版本。

### 模块不负责

- 不负责姿态关键点清洗和平滑。
- 不负责步态稳定性分析。
- 不负责近跌倒事件检测。
- 不负责个体化行为基线建模。
- 不负责最终 `risk_level`。
- 不负责 `recommended_action`。
- 不负责消息推送、家属端 App、社区端后台或可视化看板。

### 局部风险与最终风险的关系

坐站模块不输出最终 `risk_level`，但仍然可以判断“坐站这一局部能力是否异常”。

建议第一版局部解释阈值：

| `sit_stand_risk_score` | 局部含义 |
|---:|---|
| `< 0.25` | 未见明显坐站困难 |
| `0.25 - 0.50` | 轻微坐站异常 |
| `0.50 - 0.70` | 疑似坐站困难 |
| `>= 0.70` | 明显坐站困难，需融合层重点关注 |

上述分层不是最终跌倒风险等级。最终整体风险判断应交给 `FallRiskPipeline` 或后续风险融合模块：

```text
sit_stand_risk_score + gait_risk_score + near_fall_event_score + baseline_deviation_score + scene_risk_score
  -> 风险融合层
  -> risk_score + risk_level + confidence + recommended_action
```

最小示例必须支持：

```json
{
  "sit_stand_risk_score": 0.66,
  "duration": 4.8,

  "risk_factors": [
    "起身耗时较长",
    "起身后存在明显摇晃"
  ]

```

### 字段说明

| 字段 | 含义 |
|---|---|
| `transition_type` | `sit_to_stand`、`stand_to_sit` 或 `unknown_transition` |
| `sit_stand_risk_score` | 0-1 坐站局部风险分，越高表示坐站能力下降线索越强 |
| `duration` | 起身或坐下耗时，单位秒 |
| `failed_attempts` | 疑似起身失败次数 |
| `trunk_forward_angle` | 转换过程中躯干前倾角 proxy，单位度 |
| `post_stand_sway` | 起身后摇晃程度 proxy，基于归一化坐标 |
| `support_usage` | 疑似支撑使用，不代表可靠识别扶手、桌面或墙面 |
| `stabilization_time` | 起身后站稳所需时间，单位秒 |
| `sit_stand_features` | 中间特征，便于调参、复核和报告展示 |
| `quality_coverage` | 可用帧比例、关键点覆盖率、插值比例、跳变数等 |
| `risk_factors` | 可解释风险因子 |
| `model_version` | 第一版建议 `sit-stand-risk-rule-v0.1` |

低质量窗口处理原则：

- 不把低质量姿态误判为高风险。
- 当 `window_quality.usable_for_sit_stand` 不足、核心点覆盖率不足或有效帧过少时，输出 `sit_stand_risk_score = 0.0`。
- `risk_factors` 标记 `insufficient_sit_stand_quality`。
- `quality_coverage.insufficient_sit_stand_quality = true`。

## 4. Baseline 算法方案

### 4.1 预处理与分组

按 `person_id + track_id` 分组，并按 `timestamp_sec`、`frame_id` 排序。

每帧计算基础几何量：

- 肩中心：左右肩中点。
- 髋中心：左右髋中点。
- 膝中心：左右膝中点。
- 踝中心：左右踝中点。
- 身体中心：肩中心与髋中心的加权中心。
- 躯干长度 proxy：肩中心到髋中心距离。
- 腿部伸展 proxy：髋中心到踝中心的垂直距离，或髋-膝-踝夹角。
- 髋中心垂直速度。
- 髋中心横向速度。

只使用满足以下条件的点：

- `valid == true`
- `is_jump_outlier != true`
- `x_smooth/y_smooth` 可用

窗口质量门控：

- 优先读取 `window_quality.usable_for_sit_stand`。
- 统计坐站所需核心点覆盖率。
- 统计有效帧比例、插值点比例、跳变点数量。

### 4.2 坐站事件识别思路

坐标假设：当前姿态坐标通常是图像归一化坐标，`y` 越大越靠下。

#### `sit_to_stand`

疑似起身事件满足：

- 髋中心 `y_smooth` 在一段时间内整体下降。
- 肩中心 `y_smooth` 同步下降或保持合理躯干关系。
- 腿部伸展 proxy 增大。
- 膝角 proxy 从弯曲趋势转向伸展趋势。
- 转换后髋中心高度进入相对稳定状态。

第一版不依赖绝对坐姿/站姿分类，而是识别“从较低髋位到较高髋位”的连续变化。

#### `stand_to_sit`

疑似坐下事件满足：

- 髋中心 `y_smooth` 在一段时间内整体上升。
- 膝角 proxy 变小或腿部伸展 proxy 减小。
- 转换后髋中心高度进入较低稳定状态。

#### 起身失败

疑似起身失败片段满足：

- 髋中心短暂下降，表现为试图起身。
- 腿部伸展不足，未达到站立候选高度。
- 随后髋中心回到接近初始坐姿高度。
- 或者同一坐姿区域内出现多次上升尝试后才完成起身。

第一版只能输出 `failed_attempts` proxy，不应称为临床意义上的起身失败诊断。

#### 起身后摇晃

起身完成后的 1-2 秒后置窗口中，计算肩中心或髋中心横向摆动：

- 横向标准差。
- 横向最大偏移。
- 横向速度标准差。
- 躯干角变化幅度。

若后置窗口不足，应输出 `post_stand_sway = null` 或保守值，并在 `quality_coverage` 标记后置窗口不足。

#### 站稳时间

起身完成后，查找第一个稳定时间点：

- 髋中心横向速度低于阈值。
- 肩中心横向速度低于阈值。
- 躯干前倾角变化低于阈值。
- 上述条件持续若干帧。

`stabilization_time = 稳定时间点 - 起身完成时间`。

如果观察窗口内始终不稳定，则取后置观察窗口上限，并添加风险因子。

#### 疑似借助支撑

第一版只能做 proxy：

- 腕部在起身或坐下过程中长时间位于身体侧方。
- 腕部相对髋中心的距离变化很小，但身体中心明显移动。
- 腕部快速移向画面边缘或固定高度后保持。
- 单侧腕部低速稳定，另一侧身体关键点发生坐站转换。

输出格式：

```json
{
  "suspected": true,
  "evidence": [
    "left_wrist_stays_near_side_surface"
  ]
}
```

限制必须写入文档和结果说明：没有场景物体检测、深度信息或真实接触标签时，不能可靠判断扶手、桌面、墙面，只能判断“疑似支撑使用”。

### 4.3 六个关键指标计算

#### `duration`

近似计算：

- 起点：髋中心垂直速度超过阈值，且持续变化方向符合起身或坐下。
- 终点：髋中心高度和腿部伸展 proxy 进入稳定状态。
- `duration = end_time - start_time`。

限制：

- 归一化坐标不能直接对应真实世界速度。
- 正面视角下髋高度变化仍可用，但坐姿遮挡会影响稳定性。

升级：

- 引入个人基线中的常规坐站耗时分位数。
- 引入相机视角标签和人体尺度归一化。

#### `failed_attempts`

近似计算：

- 在成功 `sit_to_stand` 前，统计短暂起身趋势后回落的局部峰值。
- 以髋中心垂直位移、膝/踝伸展不足、恢复到坐姿高度作为证据。

限制：

- 遮挡、跟踪抖动、镜头正面压缩会造成误判。

升级：

- 使用动作级标注校准失败阈值。
- 结合近跌倒模块的异常下沉和手部支撑证据。

#### `trunk_forward_angle`

近似计算：

- 取肩中心到髋中心向量。
- 计算该向量相对图像竖直方向的夹角。
- 在转换窗口内取最大值或 90 分位值。

限制：

- 单目 2D 关键点无法可靠区分真实前倾和横向侧倾。
- 正面相机对前后方向不敏感。

升级：

- 引入侧面/斜侧面视角标记。
- 多摄像头或深度相机条件下再提升可靠性。

#### `post_stand_sway`

近似计算：

- 起身完成后 1-2 秒内计算髋中心或肩髋中心横向摆动。
- 可用横向标准差、最大偏移、速度标准差组合。
- 使用躯干长度或髋宽 proxy 做归一化。

限制：

- 相机移动、跟踪 ID 切换和下肢出画会显著污染该指标。

升级：

- 与步态模块共享中心稳定性计算。
- 加入轨迹跟踪质量和相机固定性检查。

#### `support_usage`

近似计算：

- 检查左右腕是否有效。
- 检查腕部是否在转换期相对身体侧方长时间稳定。
- 检查腕部是否出现“快速移动到边缘/固定区域后保持”的模式。

限制：

- 腕部关键点误检较常见。
- 没有扶手、桌面、墙面检测时，不能可靠判断真实接触。

升级：

- 增加家具/墙面/床边区域检测。
- 引入人工标注的手部支撑事件。

#### `stabilization_time`

近似计算：

- 从起身完成时刻开始。
- 找到髋中心横向速度、肩中心横向速度、躯干角速度均低于阈值且持续若干帧的时间点。
- 若一直未稳定，则用观察窗口上限，并加入风险因子。

限制：

- 后置窗口不足时不能稳定计算。
- 老人出画或被遮挡时不能当作未站稳。

升级：

- 结合连续轨迹和场景区域判断是否出画。
- 接入个体基线，比较本人历史站稳时间。

### 4.4 评分规则建议

第一版评分采用透明规则 baseline：

```text
sit_stand_risk_score =
  0.30 * duration_component
  + 0.20 * failed_attempt_component
  + 0.15 * trunk_forward_component
  + 0.15 * post_stand_sway_component
  + 0.10 * support_usage_component
  + 0.10 * stabilization_time_component
```

质量覆盖只做质量门控或降权：

- 质量不足时输出 `insufficient_sit_stand_quality`，不输出高风险。
- 可用但覆盖偏低时可增加 `reduced_sit_stand_quality_coverage` 风险因子，但权重应小，避免把遮挡当风险。

建议风险因子：

- `起身耗时较长`
- `疑似多次起身失败`
- `起身过程中躯干明显前倾`
- `起身后存在明显摇晃`
- `疑似借助支撑`
- `站稳时间较长`
- `坐站关键点质量不足`

## 6. 测试与验证计划

### 单元测试场景

必须新增以下测试：

1. 正常起身：输出 `sit_to_stand`，`sit_stand_risk_score` 较低，`duration` 合理。
2. 正常坐下：输出 `stand_to_sit`，不误判起身失败。
3. 起身耗时过长：`duration` 增大，风险分上升，包含 `起身耗时较长`。
4. 起身失败：成功起身前出现一次或多次回落，`failed_attempts > 0`。
5. 起身后摇晃：`post_stand_sway` 增大，包含 `起身后存在明显摇晃`。
6. 疑似借助支撑：腕部侧方稳定 proxy 成立，`support_usage.suspected == true`。
7. 站稳时间过长：`stabilization_time` 增大，包含 `站稳时间较长`。
8. 低质量窗口：`window_quality.usable_for_sit_stand == false` 时不误报高风险。
9. 腕部缺失：仍可计算基础坐站指标，`support_usage.suspected == false` 或证据为空。
10. 多人多轨迹：不同 `person_id + track_id` 不互相串事件。
11. JSONL 读写：`run_sit_stand_jsonl()` 能读入 cleaned pose JSONL 并写出事件 JSONL。

## 7. 风险、限制与后续升级

### 第一版限制

- 归一化 2D 坐标不能直接代表真实世界距离、速度或角度。
- 相机视角会强烈影响躯干前倾、膝角和支撑判断。
- 正面视角下，真实前倾可能在 2D 图像中不明显。
- 侧身、遮挡、低光、下肢出画会影响髋、膝、踝稳定性。
- 腕部关键点误检较常见，疑似支撑使用容易误报。
- 没有家具、墙面、扶手检测时，不能可靠判断“扶了什么”。
- 没有真实老人长期坐站风险标签，第一版阈值只能作为可解释 baseline。
- 不应把 `sit_stand_risk_score` 解读为医学诊断结论。

### 后续升级

- 采集并标注坐站动作片段：正常起身、起身困难、多次起身失败、扶物起身、坐下不稳。
- 引入个人基线：比较本人历史坐站耗时、站稳时间和摇晃程度。
- 引入相机视角标签，对正面、侧面、斜侧面使用不同阈值。
- 加入人体尺度归一化，如躯干长度、髋宽、肩宽。
- 加入家具、床边、桌面、墙面区域检测，提高支撑使用判断可靠性。
- 在有足够标签后，用轻量分类器或回归模型校准规则权重。
