# 跌倒风险数据集标注规范

更新时间：2026-06-29

本文说明跌倒风险模块中每个数据集应该怎么标、谁来标、产出什么文件。标签定义见 `docs/modules/fall_risk/data/跌倒风险标签字典.md`，本文不重复解释每个标签的含义。

## 1. 先说结论

当前项目不要把所有数据集都硬套完整人工标注。正确策略是：

```text
统一标签字典
  -> 按数据集选择可支持的标签子集
  -> 转成统一 manifest / action_labels / event_labels / risk_labels
  -> 后续训练和评估只读取统一格式
```

第一批人工标注任务，应从 LE2I/IMViA `Home_01` 开始，不要从 Fall Detection 2017、TOAGA 或 GSTRIDE 开始。

## 2. 标注文件

所有数据最终落到以下文件。公开数据集的原始目录名、txt、csv、xlsx 不能直接进入训练逻辑，必须先转换成统一标注。

| 文件 | 用途 | 谁负责 |
|---|---|---|
| `data/manifests/fall_risk_video_manifest.jsonl` | 每条视频样本的路径、数据集、人员、场景、来源标签 | 工程人员自动生成 |
| `data/annotations/fall_risk/action_labels.jsonl` | 行走、坐下、起身、拖步、近跌倒、跌倒等动作片段 | 人工标注员为主 |
| `data/annotations/fall_risk/event_labels.jsonl` | 近跌倒、跌倒、长时间静止等事件窗口 | 自动导入 + 人工复核 |
| `data/annotations/fall_risk/risk_labels.jsonl` | 0-4 级风险标签和风险因子 | 复核人员或项目负责人 |
| `data/annotations/fall_risk/annotation_review_log.jsonl` | 冲突、仲裁、修改记录 | 复核人员 |

## 3. 标注来源可信度

不同数据集的标签来源不同，不能混为同一种“真值”。

| 来源 | 可信度 | 例子 | 使用方式 |
|---|---|---|---|
| 官方逐帧/窗口标注 | 高 | LE2I txt 的跌倒开始帧和结束帧 | 可作为事件窗口评估参考 |
| 目录名或文件名标签 | 中 | `Fall/`、`ADL/`、`fall-*.mp4` | 适合作 clip-level 弱标签 |
| 人工动作标注 | 高，取决于复核 | 自采视频、LE2I 手工动作段 | 用于动作识别、坐站、近跌倒 |
| 表格风险 proxy | 中 | GSTRIDE 跌倒史、TUG、SPPB、FES-I | 用于步态风险校准，不等同真实视频事件 |
| 自动算法结果 | 低到中 | 姿态、步态、近跌倒规则输出 | 只能作特征或预标注，不能直接当人工真值 |

## 4. 第一阶段执行顺序

| 顺序 | 数据 | 任务 | 原因 |
|---:|---|---|---|
| 1 | LE2I/IMViA `Home_01/video (1).avi` | 培训示范，先人工标，后对照 txt | 居家场景、有官方跌倒窗口、样本小 |
| 2 | LE2I/IMViA `Home_01/video (2)-(6).avi` | 标注员练习 5 条 | 及时纠正时间边界和标签混淆 |
| 3 | LE2I/IMViA `Home_01` 全 30 条 | 第一批正式人工动作标注 | 数量可控，有官方 txt 可辅助复核 |
| 4 | LE2I/IMViA `Home_02`、`Coffee_room_01/02` | 扩大有监督事件集 | 同样有 txt |
| 5 | Fall Detection 2017、UR Fall | 自动导入 clip-level fall/ADL 标签，抽样人工质检 | 数量大，不适合一开始逐段手标 |
| 6 | 自采视频 | 完整动作、事件、风险三级标注 | 项目核心创新数据来源 |
| 7 | TOAGA、GSTRIDE、LTMM | 步态、长期活动和风险 proxy 整理 | 不作为普通视频动作标注任务 |

## 5. 人工标注工作流程

每条需要人工标注的视频都按同一流程做：

1. 打开视频，先完整看一遍，不标。
2. 第二遍按动作变化切片段。
3. 每个片段只从标签字典中选一个主要动作标签。
4. 填开始时间、结束时间、标签编号、备注。
5. 看不清或无法确认时标 `U01`，并写原因。
6. 标完回看一次，检查漏标、时间错位、标签混淆。
7. 提交后由复核人员抽查或双人复核。

第一次培训时不要先给标注员看官方 txt。让他先独立标，再用 txt 对照跌倒窗口，讲清楚哪里早、哪里晚、哪里把近跌倒和跌倒混了。

## 6. 各数据集标注规则

### 6.1 LE2I/IMViA

本地路径：

```text
data/external/le2i_imvia/raw/FallDataset/
```

子集状态：

| 子集 | AVI | TXT | 标注策略 |
|---|---:|---:|---|
| `Home_01` | 30 | 30 | 第一批人工教学和正式标注 |
| `Home_02` | 30 | 30 | 第二批人工标注和事件评估 |
| `Coffee_room_01` | 48 | 48 | 有监督事件评估 |
| `Coffee_room_02` | 22 | 22 | 有监督事件评估，目录名为 `Annotations_files` |
| `Lecture room` | 27 | 0 | 无监督鲁棒性测试或人工补标池 |
| `Office` | 33 | 0 | 无监督鲁棒性测试或人工补标池 |

原始 txt 规则：

- 第 1 行：跌倒开始帧。
- 第 2 行：跌倒结束帧。
- 后续行：人体框。

人工标注使用标签：

```text
A01-A04
B01-B06
C01-C05
D01-D04
U01
```

自动导入规则：

| 原始信息 | 目标文件 | 目标标签 |
|---|---|---|
| txt 第 1-2 行跌倒帧 | `event_labels.jsonl` | `event_type = fall` |
| txt 后续人体框 | 检测评估辅助文件 | 不写入动作标签 |
| 无 txt 的 `Lecture room/Office` | manifest | `label_source = unlabeled` |

注意事项：

- `Home_01/video (1).avi` 是第一条教学样本。
- 自动 txt 只能提供跌倒窗口，不能替代动作级人工标注。
- `Lecture room` 和 `Office` 没有官方 txt，不要放进有监督指标。

### 6.2 Fall Detection 2017

本地路径：

```text
data/external/fall_detection_2017/raw/VideoDataset/
```

标注策略：

| 路径 | 标签来源 | 目标标签 | 人工任务 |
|---|---|---|---|
| `VideoDataset/Fall/**/*.mp4` | 目录名 | `event_label = fall_event`，`risk_level = 4` | 抽样质检，不做第一批逐段手标 |
| `VideoDataset/ADL/**/*.mp4` | 目录名 | `event_label = non_fall_adl`，`risk_level = 0/1` | 抽样质检误报类型 |

禁止事项：

- 不要把 `timelapse*.mp4` 当训练视频。
- 不要把 `all_csvs/`、`all_plots/` 当视频样本。
- 不要按目录建样本；必须按实际 MP4 文件建样本。
- 不要把目录标签当成精确跌倒开始/结束时间。

用途：

- fall/ADL 事件二分类训练和评估。
- ADL 负样本误报分析。
- 不用于个体化行为基线和长期风险等级证明。

### 6.3 UR Fall

本地路径：

```text
data/external/ur_fall/raw/
```

标注策略：

| 文件模式 | 目标标签 | 人工任务 |
|---|---|---|
| `fall-*-cam0.mp4` | `fall_event` | 抽样检查 |
| `fall-*-cam1.mp4` | `fall_event` | 作为额外视角鲁棒性分析 |
| `adl-*-cam0.mp4` | `non_fall_adl` | 抽样检查误报 |
| `fall-*-acc.csv`、`adl-*-acc.csv` | 加速度序列 | 工程人员整理，不作为人工视频标注任务 |
| `fall-*-data.csv`、`adl-*-data.csv` | 同步/摘要数据 | 工程人员整理 |

注意事项：

- 主评估建议先只用 cam0，避免 fall 有 cam0/cam1 而 ADL 只有 cam0 造成偏差。
- UR Fall 可做交叉验证和视频 + IMU 扩展，不单独支撑最终结论。
- 当前不把 PNG 图片序列 zip 作为第一阶段必需输入。

### 6.4 TOAGA

本地路径：

```text
data/external/toaga/raw/
```

TOAGA 主要是步行视频和姿态 CSV，不是跌倒事件数据。

人工视频标注只允许使用：

```text
A01
B01
B02
B03
B04
U01
```

结构化标签由工程人员从参与者表整理：

| 字段 | 目标用途 |
|---|---|
| `Number of falls in the last 6 months` | `fall_history` proxy |
| `TUG (s)` | 行动能力风险 proxy |
| `BBS`、`POMA-Balance`、`POMA-Gait` | 平衡和步态能力 proxy |

禁止事项：

- 不要标 `fall_event`，除非视频中确实出现明确跌倒。
- 姿态 CSV 由工程人员整理，不作为人工视频标注任务。

### 6.5 GSTRIDE

本地路径：

```text
data/external/gstride/raw/GSTRIDE_database/
```

GSTRIDE 没有普通视频，不作为人工视频标注任务。

工程人员整理以下结构化标签：

| 字段 | 目标标签或 proxy |
|---|---|
| `Falls during the last year - prior to test (YES/NO)` | `fall_history` |
| `Frailty assessment (index)` | `frailty_score` / `frailty_high` |
| `SPPB assessment (index)` | `sppb_score` / `sppb_low` |
| `Time duration TUG test (s)` | `tug_seconds` / `tug_slow` |
| `Short FES-I assessment (index)` | `fear_of_falling_score` |
| gait parameter CSV | 步态速度、步幅、变异系数等特征 |

用途：

- 校准步态风险分数。
- 做跌倒史、虚弱、TUG/SPPB/FES-I 的小样本监督实验。

禁止事项：

- 不要把 GSTRIDE 当作视频跌倒事件数据。
- 不要用它验证人体检测、跟踪、姿态提取链路。

### 6.6 LTMM

本地路径：

```text
data/external/ltmm/raw/
```

LTMM 是 PhysioNet 的长期运动监测数据，主要包含腰部加速度计长时记录、实验室步行记录、临床/人口学表和居家监测报告，不是普通 RGB 视频数据。

工程入口：

```bash
conda run -n eldercare-ai python scripts/annotation/build_ltmm_manifest.py \
  --raw-dir data/external/ltmm/raw \
  --output data/manifests/ltmm_manifest.jsonl
```

标注/整理策略：

| 文件或记录 | 目标用途 | 人工任务 |
|---|---|---|
| `ClinicalDemogData_COFL.xlsx` | 跌倒史、BBS、DGI、FSST、TUG 等风险 proxy | 工程人员整理字段，不做视频标注 |
| `ReportHome75h.xlsx` | 居家长期运动摘要 | 工程人员整理活动量、节律、低活动等特征 |
| `CO*.dat`、`FL*.dat` | 长期腰部加速度计序列 | 按人生成长期活动和个体基线特征 |
| `LabWalks/*.dat` | 实验室步行加速度序列 | 步态特征校准和算法对照 |

目标标签或 proxy：

| 字段或来源 | 目标标签或 proxy |
|---|---|
| `CO*` / `FL*` 分组 | control / faller 弱标签 |
| 跌倒史字段 | `fall_history` |
| `BBS` | `balance_score` / `balance_low` |
| `DGI` | `dynamic_gait_index` / `dynamic_gait_low` |
| `FSST` | `four_square_step_seconds` / `fsst_slow` |
| `TUG` | `tug_seconds` / `tug_slow` |
| 居家监测摘要 | `activity_volume`、`sedentary_or_low_activity_proxy`、`baseline_activity_pattern` |

用途：

- 支撑长期运动监测和个体化行为基线建模。
- 与 GSTRIDE 互补，用跌倒史、平衡量表和 TUG 等 proxy 校准风险分。
- 验证融合层是否能消费“非视频长期行为特征”。

禁止事项：

- 不要把 LTMM 当作视频动作标注数据。
- 不要用 LTMM 验证人体检测、跟踪或姿态关键点链路。
- 不要把 `CO/FL` 分组直接解释为精确事件窗口；它只能作为参与者级或记录级弱标签。

### 6.7 自采视频

自采视频是坐站、近跌倒、个体化基线和最终风险融合的核心数据。

命名规则：

```text
{scene}_{subject_id}_{action_id}_{view}_{take_id}.mp4
```

示例：

```text
livingroom_p01_B03_side_t01.mp4
bedroom_p03_C01_front_t04.mp4
bathroomdoor_p02_C03_oblique_t02.mp4
```

必须标注：

| 文件 | 内容 |
|---|---|
| `action_labels.jsonl` | 完整动作片段 |
| `event_labels.jsonl` | 近跌倒、跌倒、长时间静止等异常事件 |
| `risk_labels.jsonl` | 0-4 级风险标签，由复核人员确认 |
| `subject_profiles.json` | 参与者元数据 |
| `annotation_review_log.jsonl` | 双人标注和仲裁记录 |

使用标签：

```text
A01-A04
B01-B06
C01-C05
D01-D04
U01
```

## 7. manifest 字段

建议每个视频样本写入：

```json
{
  "dataset": "le2i_imvia",
  "video_id": "le2i_home_01_video_1",
  "path": "data/external/le2i_imvia/raw/FallDataset/Home_01/Videos/video (1).avi",
  "subject_id": "unknown",
  "scene_region": "home",
  "view": "fixed_camera",
  "label_source": "le2i_txt_and_manual",
  "event_label": "fall_event",
  "risk_level": 4,
  "split_group": "le2i_home_01_video_1"
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---|---|
| `dataset` | 是 | `le2i_imvia`、`fall_detection_2017`、`ur_fall`、`toaga`、`gstride`、`self_collected` |
| `video_id` | 是 | 全局唯一，不含空格 |
| `path` | 是 | 原始视频或结构化数据路径 |
| `subject_id` | 是 | 不知道则填 `unknown`，不要空着 |
| `scene_region` | 是 | `home`、`lab`、`walking_lab`、`livingroom` 等 |
| `view` | 否 | `front`、`side`、`top`、`bottom`、`cam0`、`cam1`、`fixed_camera` |
| `label_source` | 是 | `manual`、`le2i_txt`、`filename`、`directory`、`table_proxy`、`unlabeled` 等 |
| `event_label` | 否 | `fall_event`、`non_fall_adl`、`walking`、`unknown` |
| `risk_level` | 否 | 0-4；没有依据就不填或填 `null` |
| `split_group` | 是 | 按人员或样本组划分 train/val/test，防止数据泄漏 |

## 8. 复核规则

第一批人工标注必须做复核，尤其是标注员刚开始参与项目时。

| 场景 | 复核要求 |
|---|---|
| 标注员培训前 10 条 | 100% 复核，逐条讲解 |
| 正式批量前 10% | 双人标注，统计一致率 |
| 跌倒、近跌倒、起身失败 | 全量复核 |
| `U01` 超过 20% 的视频 | 复核是否视频质量太差或标注员尚未熟悉规则 |
| 冲突样本 | 写入 `annotation_review_log.jsonl` |

冲突处理优先级：

1. 时间边界偏差小于 0.5 秒，复核人员统一修正。
2. 标签类别冲突，例如 `C03` vs `D01`，必须回看视频后仲裁。
3. 是否看得清冲突，例如正常动作 vs `U01`，以可观察证据为准。
4. 仍无法判断时保留 `U01/uncertain`，不要为了凑数据强行判定。

## 9. 数据划分规则

- 训练、验证、测试必须按人或样本组划分，不能随机按窗口划分。
- Fall Detection 2017 按 `SBJ_xx` 分组。
- UR Fall 按编号分组，cam0/cam1 同编号不能分别进入训练和测试。
- TOAGA 按 `OAWxx` 分组。
- GSTRIDE 按 `Vxxx` 参与者分组。
- 自采数据按 `subject_id` 分组。

## 10. 禁止写法

| 禁止写法 | 为什么错 |
|---|---|
| “公开跌倒数据已经证明真实老人长期风险预警有效” | 公开数据主要是实验/模拟场景，不能证明长期风险 |
| “所有数据集都使用全套 A/B/C/D 标签” | GSTRIDE 无视频，TOAGA 无跌倒事件，Fall Detection 2017 多数只有 clip-level 标签 |
| “目录是 Fall，所以精确跌倒时间就是整段视频” | 目录标签不是事件起止时间 |
| “看不清但像跌倒，所以标跌倒” | 不确定必须标 `U01/uncertain` |
| “走得慢，所以直接标高风险” | 标注员只标动作事实，风险级由复核或模型融合 |
