# 跌倒风险数据集使用、标注与训练方案

本文把当前本地数据集落到可执行研发流程：每个数据集具体用于哪一层算法、如何生成标注、如何切分、如何抽特征、如何训练和评估。

## 0. 先说清楚的限制

当前仓库已经有视频到姿态、姿态质控、步态、坐站、近跌倒和最终规则融合的推理脚本，但还没有完整的训练脚本和评估脚本。也就是说，现在可以跑规则 baseline 和生成训练特征；要训练 Logistic、LightGBM、Isolation Forest 或 TCN，还需要补 `scripts/train/` 和 `scripts/evaluate/`。

公开数据不能直接证明“真实老人长期跌倒风险预警”达标。它们主要用于三件事：

- 验证视频跌倒/ADL 事件识别链路。
- 验证姿态、质控和步态特征是否稳定。
- 为最终融合模型提供公开数据上的辅助校准和对照实验。

个体化基线、坐站能力风险、真实场景风险等级，必须依赖自采连续数据和人工风险级标注。

## 1.1 本地解压检查结果

本次检查时间：2026-06-27。当前 `data/external` 约 55G，其中 LE2I/IMViA 约 34G、UR Fall 约 7.4G、Fall Detection 2017 约 4.9G、TOAGA 约 4.5G、GSTRIDE 约 4.2G。

检查结论：

| 数据集 | 可用性 | 发现的问题 | 处理建议 |
|---|---|---|---|
| LE2I/IMViA | 190 个 AVI 均可被 `ffprobe` 读取；Home/Coffee 四个子集视频与 txt 标注一一匹配 | 原 `data/external/le2i_imvia/sample/` 已不存在；`Coffee_room_02` 标注目录拼写为 `Annotations_files`；`Lecture room` 和 `Office` 只有 AVI 没有 txt 标注 | 烟测路径改用 `raw/FallDataset/Home_01/Videos/video (1).avi`；标注导入脚本要兼容 `Annotation_files` 和 `Annotations_files`；Lecture/Office 只做无监督/人工补标 |
| Fall Detection 2017 | 所有 MP4 均可被 `ffprobe` 读取 | 16 个 Fall 子目录只有 `metadata.json` 没有 MP4；5 个 Fall 子目录和 1 个 ADL 子目录包含多个 MP4；根目录和 Fall 目录有 timelapse 视频 | manifest 必须按实际 MP4 文件建样本，不按 metadata 目录建样本；排除 `timelapse*.mp4`、`all_csvs/`、`all_plots/` |
| UR Fall | 100 个 MP4 均可被 `ffprobe` 读取；30 个 fall cam0、30 个 fall cam1、40 个 ADL cam0 均完整；acc/data CSV 未缺失 | 139 个可选 PNG 序列 zip 仍在 raw 根目录，未见解压出的图片序列目录；也不是完整 200 个 zip | 当前 MP4/CSV 路线不受影响；不要把图片序列写进第一阶段必需输入 |
| TOAGA | 28 个 MP4 均可被 `ffprobe` 读取；394 个姿态 CSV 完整 | 无明显结构问题 | 可直接用于步态链路和姿态质量对比 |
| GSTRIDE | 158 个 IMU、158 个 gait parameter、158 个 segmentation CSV 完整 | Excel 读取需要 `openpyxl`，当前环境缺该依赖 | 训练前补 `openpyxl` 或先把 xlsx 转 CSV |

另外，解压后存在 `.DS_Store`、原始 zip 和 `__MACOSX` 风格文件风险。构建 manifest 时统一过滤隐藏文件、`__MACOSX`、`.DS_Store`、`._*`、zip 和 timelapse 文件。

## 2. 统一标注文件

所有数据集最后都转换成项目统一标注，不要让训练代码直接理解每个公开数据集的原始目录。

### 2.1 `video_manifest.jsonl`

建议新增：

```text
data/manifests/fall_risk_video_manifest.jsonl
```

每行一条视频或一个可训练样本源：

```json
{
  "dataset": "ur_fall",
  "video_id": "ur_fall_fall_01_cam0",
  "path": "data/external/ur_fall/raw/fall-01-cam0.mp4",
  "subject_id": "ur_fall_01",
  "scene_region": "lab",
  "view": "cam0",
  "label_source": "filename",
  "event_label": "fall_event",
  "risk_level": 4,
  "license": "CC-BY-NC-SA-4.0",
  "split_group": "ur_fall_01"
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---|---|
| `dataset` | 是 | `le2i_imvia`、`fall_detection_2017`、`ur_fall`、`toaga`、`self_collected` |
| `video_id` | 是 | 全局唯一 ID，不要含空格 |
| `path` | 是 | 视频路径；zip 内视频可先解压到临时目录再写实际路径 |
| `subject_id` | 是 | 按人员划分 train/val/test 的关键字段 |
| `scene_region` | 是 | `home`、`lab`、`walking_lab`、`livingroom` 等 |
| `view` | 否 | `front`、`side`、`top`、`bottom`、`cam0`、`cam1` |
| `event_label` | 否 | `fall_event`、`non_fall_adl`、`walking`、`unknown` |
| `risk_level` | 否 | 0-4；公开事件数据中 fall 可标 4，ADL 可标 0/1 |
| `split_group` | 是 | 一般等于 subject_id，防止同一人泄漏到不同集合 |

### 2.4 风险级标注 `risk_labels.jsonl`

用于最终融合模型训练：

```json
{
  "video_id": "corridor_p02_C02_oblique_t03",
  "subject_id": "p02",
  "start_time": 0.0,
  "end_time": 9.8,
  "risk_level": 3,
  "risk_score": 0.82,
  "risk_factors": ["wall_support_walk", "gait_instability"],
  "label_source": "manual_consensus",
  "review_status": "final"
}
```

风险等级规则：

| 等级 | 可观察证据 | 下游建议 |
|---:|---|---|
| 0 | 正常行走、正常坐站、无异常停留 | `record_only` |
| 1 | 轻微慢走、轻微摇晃、短暂停顿 | `observe` |
| 2 | 拖步、小碎步、起身慢、转身不稳 | `remind_user` |
| 3 | 近跌倒、扶墙恢复、明显失衡、起身失败 | `notify_guardian` |
| 4 | 跌倒、倒地静止、疑似无法起身 | `emergency_alert` |

## 3. 各数据集如何标注

### 3.1 LE2I/IMViA

原始标注：

- 第 1 行：跌倒开始帧。
- 第 2 行：跌倒结束帧。
- 后续行：每帧人体框。

当前完整解压路径：

```text
data/external/le2i_imvia/raw/FallDataset/Home_01/Videos/video (1).avi
data/external/le2i_imvia/raw/FallDataset/Home_01/Annotation_files/video (1).txt
```

子集状态：

| 子集 | AVI | TXT | 状态 |
|---|---:|---:|---|
| `Coffee_room_01` | 48 | 48 | 可做监督评估 |
| `Coffee_room_02` | 22 | 22 | 可做监督评估；标注目录名是 `Annotations_files` |
| `Home_01` | 30 | 30 | 可做监督评估和烟测 |
| `Home_02` | 30 | 30 | 可做监督评估 |
| `Lecture room` | 27 | 0 | 只能做无监督/人工补标 |
| `Office` | 33 | 0 | 只能做无监督/人工补标 |

`video (1).txt` 中跌倒帧为 `144-164`。该视频约 24 FPS，所以事件时间约为：

```text
start_time = 144 / 24 = 6.00s
end_time   = 164 / 24 = 6.83s
```

应生成两类标注：

1. `event_labels.jsonl`：跌倒窗口。
2. `risk_labels.jsonl`：整段或跌倒窗口标 4 级，跌倒前窗口可用于近跌倒/预警误差分析。

推荐用法：

- 固定烟测：每次修改视觉链路后跑 `Home_01/Videos/video (1).avi`。
- 检测/跟踪评估：用 txt 后续人体框和 YOLO/ByteTrack 框计算 IoU、漏检率、ID 稳定性。
- 近跌倒调试：检查算法是否在跌倒开始前 0.5-2 秒内给出高分。
- 弱监督扩展：`Lecture room` 和 `Office` 不进入有监督指标，只能作为无标签视频做姿态质量、模型鲁棒性或人工补标池。

manifest 导入注意：

- 同时识别 `Annotation_files` 和 `Annotations_files`。
- `Lecture room`、`Office` 下视频直接位于子集目录，不在 `Videos/` 子目录。
- 旧的 `data/external/le2i_imvia/sample/` 已不存在，不要再写入新命令或新 manifest。

### 3.2 Fall Detection 2017

解压后路径：

```text
data/external/fall_detection_2017/raw/VideoDataset/Fall/SBJ_xx_LOCy/.../*.mp4
data/external/fall_detection_2017/raw/VideoDataset/ADL/SBJ_xx_LOCy/.../*.mp4
```

标注规则：

| 路径 | `event_label` | `risk_level` | 用途 |
|---|---|---:|---|
| `VideoDataset/Fall/SBJ_*/**/*.mp4` | `fall_event` | 4 | 跌倒事件正样本 |
| `VideoDataset/ADL/SBJ_*/**/*.mp4` | `non_fall_adl` | 0/1 | 非跌倒负样本 |

不要把 `timelapse*.mp4`、`all_plots/`、`all_csvs/` 当训练视频。

本地检查发现：

- Fall 事件 MP4：997 个。
- ADL 事件 MP4：1017 个。
- Fall/ADL 各有 21 个 `SBJ_*` 目录。
- Fall 中有 16 个子目录只有 `metadata.json` 没有 MP4，不能建样本。
- Fall 中有 5 个子目录、ADL 中有 1 个子目录包含多个 MP4，应按每个 MP4 建样本，而不是按目录建样本。
- 所有 MP4 可被 `ffprobe` 读取。

切分规则：

- 按 `SBJ_xx` 分组，不能随机按片段切。
- 推荐：`SBJ_01-SBJ_15` 训练，`SBJ_16-SBJ_18` 验证，`SBJ_19+` 测试；实际要检查各集合 Fall/ADL 数量是否平衡。

训练用途：

- 第一优先：训练或评估 `fall_event_score`。
- 第二优先：作为 ADL 负样本，降低近跌倒误报。
- 不建议：训练个体化基线或长期风险趋势。

样本构造：

1. 每个 MP4 跑姿态和质控。
2. 用 1.5-2 秒滑窗生成窗口级特征。
3. 如果窗口来自 `Fall/`，且靠近跌倒动作核心段，标 `fall_event=1`；如果没有准确事件时间，先把整段视频标为 clip-level 正样本，训练 clip classifier，报告中说明这是弱标签。
4. 如果窗口来自 `ADL/`，标 `fall_event=0`。
5. 如果同一目录有多个 MP4，每个 MP4 使用文件名追加到 `video_id`，例如 `fall2017_sbj10_loc3_act10_f_2_20240915185702`。

### 3.3 UR Fall

当前本地核心文件完整：

| 类型 | 数量 | 说明 |
|---|---:|---|
| `fall-*-cam0.mp4` | 30 | 跌倒 camera 0 |
| `fall-*-cam1.mp4` | 30 | 跌倒 camera 1 |
| `adl-*-cam0.mp4` | 40 | ADL camera 0 |
| `fall-*-acc.csv` | 30 | 跌倒加速度 |
| `adl-*-acc.csv` | 40 | ADL 加速度 |
| `fall-*-data.csv` | 30 | 跌倒同步/摘要 |
| `adl-*-data.csv` | 40 | ADL 同步/摘要 |

所有 100 个 MP4 可被 `ffprobe` 读取。当前仍有 139 个可选深度/RGB PNG 序列 zip 位于 `data/external/ur_fall/raw/`，但没有解压出的图片序列目录；这不影响第一阶段 MP4/CSV 工作流。

文件规则：

| 文件 | 标签 |
|---|---|
| `fall-*-cam0.mp4` | `fall_event` |
| `fall-*-cam1.mp4` | `fall_event` |
| `adl-*-cam0.mp4` | `non_fall_adl` |
| `fall-*-acc.csv` | 跌倒对应加速度序列 |
| `adl-*-acc.csv` | ADL 对应加速度序列 |
| `fall-*-data.csv` / `adl-*-data.csv` | 同步帧/传感器摘要 |
| `urfall-cam0-falls.csv` / `urfall-cam0-adls.csv` | 官方已提取特征，可做对照 |

切分规则：

- 按编号切，例如 `01-20` 训练，`21-25` 验证，`26-30/40` 测试。
- 注意 Fall 有 cam0/cam1，ADL 只有 cam0。评估主表建议只用 cam0；cam1 作为额外视角鲁棒性分析。

训练用途：

- 用 MP4 训练/评估 `fall_event_score`。
- 用 `acc.csv` 做后续多模态融合：视频模型输出 + 加速度峰值/方差/姿态变化。
- 用 ADL 负样本检查坐下、弯腰、下蹲是否误报成跌倒。

不要做：

- 不要把 UR Fall 当商业可用数据。
- 不要用它独立支撑最终比赛结论。
- 不要把 PNG 图片序列作为第一阶段必需输入；当前本地没有完整 200 个图片序列 zip，也没有展开后的图片目录。

### 3.4 TOAGA

视频：

```text
data/external/toaga/raw/Videos/OAW01-top.mp4
data/external/toaga/raw/Videos/OAW01-bottom.mp4
...
```

姿态 CSV：

```text
data/external/toaga/raw/Pose Tracking/Alphapose/01/*.csv
data/external/toaga/raw/Pose Tracking/OpenPose/01/*.csv
data/external/toaga/raw/Pose Tracking/Detectron/01/*.csv
```

参与者表字段包括：

```text
Participant No.
Age (years)
Sex
Height (cm)
Weight (lbs)
Number of falls in the last 6 months
POMA-Balance
POMA-Gait
BBS
TUG (s)
```

标注规则：

| 来源 | 标签 |
|---|---|
| 所有 `OAWxx-*.mp4` | `event_label = walking` |
| `Number of falls in the last 6 months > 0` | `fall_history = 1` |
| `TUG (s)` 高 | `mobility_risk_proxy = 1`，阈值需在报告中说明 |
| `BBS` 低、`POMA` 低 | 平衡能力风险 proxy |

训练用途：

- 验证视频步态特征是否能反映步态稳定性。
- 比较本项目 YOLOv8 pose 与 TOAGA 自带 AlphaPose/OpenPose/Detectron 的关键点质量。
- 用参与者功能量表做轻量监督，例如预测 `fall_history`、`TUG_high` 或 `BBS_low`。

具体流程：

1. 对 `OAWxx-top/bottom.mp4` 跑本项目姿态和 gait。
2. 把 `gait_stability_features` 聚合到视频级或参与者级。
3. 从 `Table_1.xlsx` 抽取同一个 `OAWxx` 的 `TUG`、`BBS`、`POMA`、跌倒次数。
4. 训练小模型预测 `fall_history` 或 `TUG_high`。

限制：

- TOAGA 是步行数据，不是跌倒事件数据。
- 不能用它训练 `fall_event_score`。

### 3.5 GSTRIDE

目录：

```text
data/external/gstride/raw/GSTRIDE_database/_IMU/V001.csv
data/external/gstride/raw/GSTRIDE_database/_GAIT/_SEGMENTATION/V001_GAIT_SEGMENTATION.csv
data/external/gstride/raw/GSTRIDE_database/_GAIT/_PARAMETERS/V001_GAIT_PARAMETERS.csv
data/external/gstride/raw/GSTRIDE_database/GSTRIDE_DDBB.xlsx
```

核心字段：

```text
Falls during the last year - prior to test (YES/NO)
Frailty assessment (index)
SPPB assessment (index)
Time duration TUG test (s)
Short FES-I assessment (index)
Gait speed (m/s)
mean_StrideLength[m]
mean_Stride_Velocity[m/s]
std_Stride_Velocity[m/s]
cv_Stride_Velocity[m/s]
cv_CycleDuration[s]
```

训练用途：

- 训练步态风险校准器：`gait_risk_score` 不再只靠规则阈值，而是用临床/功能 proxy 校准。
- 训练目标可以是：
  - `fall_history`: 过去一年是否跌倒。
  - `frailty_high`: 虚弱评分是否高。
  - `sppb_low`: SPPB 是否低。
  - `tug_slow`: TUG 是否慢。
  - `fes_i_high`: 跌倒恐惧是否高。

输入特征：

```text
mean_CycleDuration[s]
mean_Cadence[steps/min]
mean_StrideLength[m]
mean_Clearance[m]
mean_Stride_Velocity[m/s]
std_Stride_Velocity[m/s]
cv_CycleDuration[s]
cv_StrideLength[m]
cv_Stride_Velocity[m/s]
```

模型：

- 样本量只有 158，优先 Logistic Regression、Random Forest、LightGBM 小树深。
- 必须做交叉验证，不要只报一次随机切分。

限制：

- GSTRIDE 没有视频，不能验证 `video -> detection -> pose -> gait`。
- 它只能校准 gait/risk 关系，不能替代视频算法训练。

### 3.6 自采数据

这是最终风险融合最关键的数据。

命名规则：

```text
{scene}_{subject_id}_{action_id}_{view}_{take_id}.mp4
```

示例：

```text
livingroom_p01_B03_side_t01.mp4
bedroom_p03_C06_front_t04.mp4
bathroomdoor_p02_D04_oblique_t02.mp4
```

必须人工标注：

- `action_labels.jsonl`：整段或动作片段。
- `event_labels.jsonl`：只标异常事件窗口。
- `risk_labels.jsonl`：最终风险等级。
- `subject_profiles.json`：参与者元数据。
- `annotation_review_log.jsonl`：复核记录。

训练用途：

| 模型 | 自采数据作用 |
|---|---|
| 动作识别 | 主训练数据 |
| 坐站能力 | 主训练数据 |
| 近跌倒 | 主训练和阈值校准 |
| 个体化基线 | 唯一可靠数据来源 |
| 风险融合 | 主监督标签来源 |

## 5. 训练任务拆解

### 5.1 任务 A：跌倒/ADL 事件模型

目标字段：

```text
fall_event_score
```

训练数据：

- 正样本：Fall Detection 2017 的 `Fall/`、UR Fall 的 `fall-*.mp4`、LE2I 的跌倒窗口。
- 负样本：Fall Detection 2017 的 `ADL/`、UR Fall 的 `adl-*.mp4`、自采正常动作。

输入特征：

- `near_fall_features.body_center_lateral_velocity_peak`
- `near_fall_features.body_center_lateral_acceleration_peak`
- `near_fall_features.hip_vertical_drop`
- `near_fall_features.hip_recovery_ratio`
- `near_fall_features.trunk_angle_delta_deg`
- `near_fall_features.leg_compression_delta`
- `gait_stability_features.center_speed_cv`
- `quality_coverage.mean_core_keypoint_quality`

第一版模型：

- Logistic Regression 或 Random Forest。
- 训练目标是二分类：`fall_event = 1/0`。
- 输出概率作为 `fall_event_score`。

评估：

- Clip-level Accuracy、Precision、Recall、F1、AUC。
- 重点看 Recall 和 ADL 误报。
- 对跌倒样本报告首次触发时间。

### 5.2 任务 B：近跌倒模型

目标字段：

```text
near_fall_event_score
```

训练数据：

- 正样本：自采 `C03/C04/C05`，LE2I 跌倒前 0.5-2 秒窗口可作为弱正样本。
- 负样本：Fall Detection 2017 ADL、UR ADL、自采 A/B 类正常和轻风险动作。

第一版训练策略：

1. 规则 baseline 继续保留。
2. 用 ADL/正常数据训练 Isolation Forest，学习正常姿态窗口分布。
3. 用自采近跌倒事件做阈值校准。

输入特征：

- 横向速度峰值。
- 横向加速度峰值。
- 髋部快速下沉。
- 躯干角变化。
- 急停比例。
- 腿部压缩变化。
- 手腕疑似支撑 proxy。

输出：

- 规则分数和异常检测分数组合为 `near_fall_event_score`。
- 强触发时 `event_type` 应为 `stumble_or_lateral_loss`、`rapid_descent_recovery` 或 `support_contact_proxy`。

### 5.3 任务 C：步态风险校准模型

目标字段：

```text
gait_risk_score
```

训练数据：

- TOAGA：视频步态特征 + 参与者表中的跌倒次数、TUG、BBS、POMA。
- GSTRIDE：IMU gait parameter + 跌倒史、frailty、SPPB、TUG、FES-I。
- 自采：正常行走、慢走、拖步、小碎步、摇晃行走。

训练方式分两层：

1. 视频步态特征层：用 TOAGA 和自采视频训练/调规则。
2. 临床 proxy 校准层：用 GSTRIDE 学习 gait metrics 与 fall/frailty 的关系。

推荐标签：

```text
gait_risk_label = 1 if fall_history == YES or frailty_high or sppb_low or tug_slow else 0
```

注意：阈值必须在报告中固定并说明，不要调到测试集最好看。

模型：

- 小样本：Logistic Regression。
- 非线性对照：LightGBM，限制 `num_leaves` 和 `max_depth`。

评估：

- 按参与者分组交叉验证。
- 报 AUC、F1、Recall。
- 报特征重要性：步速、步幅、步态变异系数等。

### 5.4 任务 D：坐站能力模型

目标字段：

```text
sit_stand_risk_score
```

训练数据：

- 主要靠自采：A03/A04 正常坐站，B06 起身缓慢，C01 起身失败。
- 公开数据只能辅助提供姿态负样本，不能替代坐站标注。

输入特征：

- `duration`
- `failed_attempts`
- `trunk_forward_angle`
- `post_stand_sway`
- `stabilization_time`
- `support_usage.suspected`
- `sit_stand_features.hip_vertical_displacement`
- `quality_coverage.sit_stand_keypoint_coverage`

标签：

| 标签 | 来源动作 |
|---|---|
| 0 | A03/A04 正常坐站 |
| 1 | B06 起身缓慢 |
| 2 | C01 起身失败或明显借力 |

第一版：

- 先用规则分数作为 baseline。
- 数据超过 200 条坐站事件后训练 Logistic/Random Forest。
- 数据超过 500 条且动作边界稳定后，再考虑 TCN。

### 5.5 任务 E：个体化基线

目标字段：

```text
baseline_deviation_score
```

训练数据：

- 只能用连续自采数据。
- 每人至少 3 天可建立初始基线，7 天以上较稳。

标注：

```json
{
  "person_id": "p01",
  "date": "2026-07-01",
  "period": "day",
  "mean_gait_speed_proxy": 0.12,
  "mean_sit_stand_duration": 3.1,
  "near_fall_count": 0,
  "nighttime_activity_count": 1,
  "activity_volume": 0.74,
  "dominant_scene_region": "livingroom"
}
```

训练/建模：

- 当前仓库已有滚动均值、标准差、分位数规则 baseline。
- 不建议一开始训练深度时序模型。
- 先报告“去掉基线 vs 加入基线”的消融实验。

命令：


### 5.6 任务 F：最终风险融合模型

目标：

```text
risk_level
risk_score
recommended_action
```

当前规则权重：

| 特征 | 权重 |
|---|---:|
| `gait_risk_score` | 0.22 |
| `sit_stand_risk_score` | 0.18 |
| `near_fall_event_score` | 0.28 |
| `baseline_deviation_score` | 0.16 |
| `scene_risk_score` | 0.08 |
| `activity_rhythm_score` | 0.08 |

训练数据：

- 自采 `risk_labels.jsonl` 是主数据。
- 公开数据只补 `fall_event_score`、ADL 负样本和 gait proxy。

第一版模型：

- 规则评分卡作为 baseline。
- 有 300+ 条风险级标注后，训练 Logistic Regression 或 LightGBM 多分类。
- 训练目标可以是：
  - 多分类：0/1/2/3/4。
  - 或回归：预测 0-1 风险分，再映射到等级。

输入特征：

```text
gait_risk_score
sit_stand_risk_score
near_fall_event_score
baseline_deviation_score
scene_risk_score
activity_rhythm_score
fall_event_score
long_static_score
keypoint_quality
feature_coverage
```

硬规则必须保留：

- `fall_event_score >= 0.8` 或 `long_static_score >= 0.8` 直接 4 级。
- `near_fall_event_score >= 0.7` 至少 3 级。
- `feature_coverage` 或 `keypoint_quality` 太低时降置信度，必要时输出 `manual_review`。

## 7. 评估方式

### 7.1 公开事件数据

用于 `fall_event_score`：

| 指标 | 说明 |
|---|---|
| Precision | 报警里有多少真跌倒 |
| Recall | 真跌倒召回多少 |
| F1 | 综合指标 |
| AUC | 二分类排序能力 |
| False positive cases | ADL 被误判为跌倒的类型 |

### 7.2 近跌倒

用于 `near_fall_event_score`：

- 近跌倒事件召回率。
- 误报率。
- 平均首次触发时间。
- 事件起止时间 IoU 或时间重叠率。

### 7.3 风险融合

用于最终 `risk_level`：

- 0-4 等级 Accuracy。
- 高风险召回率：人工 3-4 级中算法输出 >=3 的比例。
- 高风险误报率：算法 >=3 但人工 0-1 的比例。
- 相邻等级容忍准确率：预测等级与人工等级差值 <=1。
- 消融实验：去掉基线、去掉近跌倒、去掉场景、只用规则。

### 7.4 必须按人员划分

禁止同一个参与者同时进入训练集和测试集。公开数据也要按 `SBJ_xx`、`fall-xx/adl-xx`、`OAWxx`、`Vxxx` 分组。

## 10. 报告中应该怎么写

可以写：

```text
公开跌倒/ADL 数据集用于验证跌倒事件识别和误报控制；
TOAGA 用于验证视频姿态步态链路；

```

不要写：

```text
仅凭公开跌倒数据集已经完成真实老人长期跌倒风险预警模型训练。
```

这个说法证据不足，评审追问时会站不住。
