# 跌倒风险算法服务后端对接说明

## 1. 文档目的

本文档用于系统后端与跌倒风险算法服务联调，只约定以下两部分：

1. 系统后端提供一个风险事件回调接口，供算法服务上报风险事件。
2. 系统后端调用算法服务的 HTTP 接口，启动、查询、更新和停止监测会话。

系统后端自行负责事件持久化、前端接口、WebSocket、消息通知和后续处置。算法服务不访问系统数据库，也不实现账号、设备管理或通知流程。

## 2. 对接关系

```text
系统后端                              跌倒风险算法服务
   |                                        |
   |  POST /v1/monitoring/sessions          |
   |  提交直播地址和 callback_url            |
   | -------------------------------------> |
   |                                        |
   |  POST callback_url                     |
   |  上报风险事件                           |
   | <------------------------------------- |
   |                                        |
   |  GET / PUT / stop                      |
   |  查询、刷新直播地址或停止会话             |
   | -------------------------------------> |
```

双方需要在部署时约定以下配置：

| 配置 | 使用方向 | 说明 |
|---|---|---|
| `ALGORITHM_BASE_URL` | 后端 -> 算法 | 算法服务地址，例如 `http://algorithm-service:8080` |
| `ALGORITHM_API_TOKEN` | 后端 -> 算法 | 后端调用算法接口时使用的 Bearer Token |
| `CALLBACK_TOKEN` | 算法 -> 后端 | 算法调用风险回调接口时使用的 Bearer Token |
| `callback_url` | 算法 -> 后端 | 后端提供的完整回调 URL，必须能从算法容器访问 |

`ALGORITHM_API_TOKEN` 和 `CALLBACK_TOKEN` 用途不同，不应混用。Token 通过部署配置或密钥管理工具传递，不写入代码仓库或日志。

## 第一部分：风险事件回调

## 3. 后端需要提供的接口

接口路径由系统后端决定。以下路径仅作为示例：

```http
POST /api/algorithm-events
Authorization: Bearer ${CALLBACK_TOKEN}
Content-Type: application/json
```

后端在启动算法会话时，通过 `callback_url` 将该接口的完整 URL 传给算法服务。

### 3.1 回调请求示例

```json
{
  "event_id": "f5f4883c-3f42-46e9-9db5-dbe2bb0040d0",
  "session_id": "b7dd2db3-3877-44fd-a510-e19821f8597d",
  "schema_version": "1.0",
  "module": "fall_risk",
  "device_id": "cam_001",
  "person_id": "elder_001",
  "timestamp": "2026-07-12T14:20:00+08:00",
  "scene_region": "living_room",
  "risk_level": 3,
  "risk_score": 0.84,
  "confidence": 0.87,
  "trigger_event": "near_fall",
  "risk_factors": [
    "near_fall_event",
    "gait_instability"
  ],
  "recommended_action": "notify_guardian",
  "evidence_window": {
    "start_time": 15.2,
    "end_time": 17.6
  },
  "model_version": "fall-risk-v0.1",
  "metadata": {}
}
```

### 3.2 回调字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `event_id` | string | 风险事件唯一标识。算法重试同一事件时保持不变，后端以此字段幂等去重 |
| `session_id` | string | 算法监测会话 ID，与启动会话返回值一致 |
| `schema_version` | string | 回调结构版本，当前固定为 `1.0` |
| `module` | string | 模块编码，当前实时服务固定为 `fall_risk` |
| `device_id` | string | 摄像头或视频设备编号，来自启动请求 |
| `person_id` | string | 被监测人员编号，来自启动请求 |
| `timestamp` | string | 事件生成时间，带时区的 ISO 8601 时间 |
| `scene_region` | string | 场景区域编码，来自启动请求 |
| `risk_level` | integer | 风险等级，实时回调只发送 1-4 级事件 |
| `risk_score` | number | 综合风险分数，预期范围为 0-1 |
| `confidence` | number | 输入质量和特征覆盖度形成的置信度，预期范围为 0-1，不代表医学确定性 |
| `trigger_event` | string | 主要触发类型，编码见下表 |
| `risk_factors` | string[] | 可解释风险因子编码，可为空数组 |
| `recommended_action` | string | 算法建议动作编码，不代表系统已执行该动作 |
| `evidence_window` | object 或 null | 证据时间窗，时间值为当前监测会话开始后的相对秒数 |
| `evidence_window.start_time` | number 或 null | 证据窗开始时间，单位秒 |
| `evidence_window.end_time` | number 或 null | 证据窗结束时间，单位秒 |
| `model_version` | string | 算法模型或规则版本 |
| `metadata` | object | 扩展信息，当前通常为空对象；后端应允许出现新增键 |

### 3.3 风险等级

| `risk_level` | 含义 | 默认建议动作 |
|---:|---|---|
| 0 | 正常 | `record_only`；实时服务不回调 0 级事件 |
| 1 | 低风险 | `observe` |
| 2 | 中风险 | `remind_user` |
| 3 | 高风险 | `notify_guardian` |
| 4 | 紧急风险 | `emergency_alert` |

系统后端应根据自身业务规则决定是否通知、通知对象和处置流程，不应仅依据 `recommended_action` 直接执行不可逆操作。

### 3.4 当前触发类型

| `trigger_event` | 含义 |
|---|---|
| `mild_deviation` | 轻度综合偏离 |
| `mobility_risk` | 行动能力相关风险 |
| `near_fall` | 近跌倒风险 |
| `combined_high_risk` | 多项信号融合形成的高风险 |
| `fall_or_long_static` | 疑似跌倒或跌倒后长时间静止 |

### 3.5 当前风险因子

| 风险因子编码 | 含义 |
|---|---|
| `suspected_fall_event` | 疑似跌倒事件 |
| `long_static_after_fall_risk` | 疑似跌倒后长时间静止 |
| `near_fall_event` | 近跌倒事件 |
| `gait_instability` | 步态不稳定 |
| `sit_stand_difficulty` | 坐站转换困难 |
| `personal_baseline_deviation` | 相对个人历史基线发生偏离 |
| `activity_rhythm_change` | 活动节律变化 |
| `high_risk_scene` | 当前区域属于较高风险场景 |

后续算法版本可能增加新的触发类型或风险因子。后端应将未知编码按原值保存，并使用默认展示文案，不应因为出现未知编码而拒绝整个回调请求。

## 4. 回调响应要求

后端成功校验并接收事件后，返回任意 `2xx` 状态。推荐不返回响应体：

```http
HTTP/1.1 204 No Content
```

处理规则：

- 校验 `Authorization: Bearer ${CALLBACK_TOKEN}`；Token 缺失或错误时返回 `401`。
- 使用 `event_id` 建立唯一约束或执行幂等写入。
- 同一 `event_id` 重复到达时不得重复创建事件或重复触发通知，但仍应返回 `2xx`。
- 先完成可靠接收或持久化，再返回 `2xx`；耗时通知任务建议由后端异步处理。
- 算法把网络异常和所有非 `2xx` 响应视为失败，最多发送 3 次；最终失败不会停止算法推理。
- 当前算法侧单次请求超时默认为 5 秒。

建议后端保存原始回调 JSON，至少对 `event_id`、`session_id`、`device_id`、`person_id`、`timestamp`、`risk_level` 和接收结果记录结构化日志。日志中不要记录 Bearer Token。

## 第二部分：后端调用算法服务

## 5. 通用调用约定

除健康检查外，算法接口均使用同一个请求头：

```http
Authorization: Bearer ${ALGORITHM_API_TOKEN}
Content-Type: application/json
```

算法服务当前为单路原型：单个服务进程同时最多保留一个未终止会话。需要并行处理多路摄像头时，应部署多个独立算法服务实例，并由系统侧完成实例分配。

## 6. 启动监测会话

```http
POST ${ALGORITHM_BASE_URL}/v1/monitoring/sessions
Authorization: Bearer ${ALGORITHM_API_TOKEN}
Content-Type: application/json
```

请求示例：

```json
{
  "request_id": "start-cam-001-20260712T142000",
  "stream_url": "https://example.invalid/live.m3u8",
  "device_id": "cam_001",
  "person_id": "elder_001",
  "scene_region": "living_room",
  "callback_url": "https://backend.example.invalid/api/algorithm-events"
}
```

| 字段 | 类型 | 约束 |
|---|---|---|
| `request_id` | string | 必填、非空；一次启动操作的幂等键 |
| `stream_url` | string | 必填；仅支持 `rtsp://`、`rtmp://`、`http://` 或 `https://` 可直接解码的地址 |
| `device_id` | string | 必填、非空；系统设备编号 |
| `person_id` | string | 必填、非空；系统人员编号 |
| `scene_region` | string | 必填、非空；双方约定的场景区域编码 |
| `callback_url` | string | 必填；仅支持 `http://` 或 `https://`，且算法容器必须能够访问 |

算法服务不负责将萤石 `ezopen://` 地址转换成可播放直播地址。系统后端应从萤石开放平台取得算法容器可直接解码的直播地址后再调用本接口。

当前算法配置使用以下场景区域编码：

| `scene_region` | 含义 |
|---|---|
| `living_room` | 客厅 |
| `bedroom_bedside` | 卧室床边 |
| `bathroom` | 卫生间 |
| `stairs` | 楼梯区域 |
| `home` | 未细分的普通居家区域 |

接口能够接收其他非空编码，但未配置的区域按未知区域处理，不增加场景风险分。双方增加新编码时应同步更新算法配置。

成功返回 `202 Accepted`：

```json
{
  "session_id": "b7dd2db3-3877-44fd-a510-e19821f8597d",
  "status": "starting"
}
```

后端必须保存 `session_id`，后续状态查询、地址更新和停止操作均使用它。

幂等规则：

- 相同 `request_id` 重复提交时，算法返回原会话，不创建新线程。
- 后端在请求超时且不确定算法是否已接收时，应使用原 `request_id` 重试。
- 一次会话结束后，如需重新启动，应生成新的 `request_id`。
- 已有未终止会话时，使用不同 `request_id` 启动会返回 `409 Conflict`。

curl 示例：

```bash
curl -X POST "${ALGORITHM_BASE_URL}/v1/monitoring/sessions" \
  -H "Authorization: Bearer ${ALGORITHM_API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "request_id": "start-cam-001-20260712T142000",
    "stream_url": "https://example.invalid/live.m3u8",
    "device_id": "cam_001",
    "person_id": "elder_001",
    "scene_region": "living_room",
    "callback_url": "https://backend.example.invalid/api/algorithm-events"
  }'
```

## 7. 查询会话状态

```http
GET ${ALGORITHM_BASE_URL}/v1/monitoring/sessions/{session_id}
Authorization: Bearer ${ALGORITHM_API_TOKEN}
```

成功返回 `200 OK`：

```json
{
  "session_id": "b7dd2db3-3877-44fd-a510-e19821f8597d",
  "status": "running",
  "device_id": "cam_001",
  "person_id": "elder_001",
  "started_at": "2026-07-12T06:20:00+00:00",
  "last_frame_at": "2026-07-12T06:20:03+00:00",
  "last_error": null
}
```

| 字段 | 类型 | 含义 |
|---|---|---|
| `status` | string | 当前会话状态，枚举见下表 |
| `started_at` | string | 会话创建时间，带时区的 ISO 8601 时间 |
| `last_frame_at` | string 或 null | 最近成功读取视频帧的时间；尚未出帧时为 null |
| `last_error` | string 或 null | 失败原因；仅用于运维诊断，不建议直接展示给最终用户 |

状态枚举：

| 状态 | 含义 |
|---|---|
| `starting` | 后台会话已创建，正在初始化模型或视频流 |
| `running` | 正常读取视频并运行算法 |
| `reconnecting` | 视频流中断，正在有限次数重连 |
| `stopping` | 正在停止并释放资源 |
| `stopped` | 已停止 |
| `failed` | 初始化、读取或重连失败；具体原因见 `last_error` |

推荐后端在启动后轮询至 `running`、`failed` 或业务超时。进入 `failed` 后需排查 `last_error`；重新启动时使用新的 `request_id`。

## 8. 更新直播地址

萤石直播地址过期或系统获取到新地址后，调用：

```http
PUT ${ALGORITHM_BASE_URL}/v1/monitoring/sessions/{session_id}/stream-url
Authorization: Bearer ${ALGORITHM_API_TOKEN}
Content-Type: application/json
```

```json
{
  "stream_url": "https://example.invalid/refreshed-live.m3u8"
}
```

成功返回 `200 OK`：

```json
{
  "session_id": "b7dd2db3-3877-44fd-a510-e19821f8597d",
  "status": "running"
}
```

算法会释放旧视频连接，由同一会话线程读取新地址；`session_id` 不变。更新后建议查询会话状态和 `last_frame_at`，确认新地址已正常出帧。

## 9. 停止监测会话

```http
POST ${ALGORITHM_BASE_URL}/v1/monitoring/sessions/{session_id}/stop
Authorization: Bearer ${ALGORITHM_API_TOKEN}
```

成功返回 `202 Accepted`：

```json
{
  "session_id": "b7dd2db3-3877-44fd-a510-e19821f8597d",
  "status": "stopped"
}
```

停止接口幂等。相同 `session_id` 重复停止不会创建额外操作；后端可以在网络超时后安全重试。

## 10. 健康检查

健康检查不需要鉴权：

| 方法与路径 | 成功响应 | 含义 |
|---|---|---|
| `GET /health/live` | `200 {"status":"ok"}` | 算法服务进程存活 |
| `GET /health/ready` | `200 {"status":"ready"}` | 配置已加载且模型文件存在 |

`GET /health/ready` 在模型文件不可用时返回 `503`。部署平台可使用健康检查；业务后端也可以在启动会话前按需检查就绪状态。

## 11. 算法接口错误响应

错误响应使用 FastAPI 标准 JSON 结构，通常为：

```json
{
  "detail": "错误原因"
}
```

| HTTP 状态 | 场景 | 后端建议 |
|---:|---|---|
| `401` | Token 缺失或错误 | 检查部署配置，不进行无休止重试 |
| `404` | `session_id` 不存在 | 校验实例地址和本地保存的会话 ID |
| `409` | 当前算法实例已有其他活动会话 | 停止原会话或选择其他算法实例 |
| `422` | 请求缺少字段、存在多余字段、字段为空或 URL 协议不支持 | 修正请求，不直接重试原参数 |
| `500` | 算法服务内部异常 | 记录响应和请求标识，有限重试并告警 |
| `503` | `/health/ready` 检查未就绪 | 检查模型挂载和服务配置 |

## 12. 推荐联调流程

1. 双方配置两个独立 Token，并确认算法服务能访问后端 `callback_url`。
2. 后端调用 `/health/live` 和 `/health/ready`。
3. 后端使用可直接解码的短视频流地址创建会话并保存 `session_id`。
4. 后端查询会话，确认状态进入 `running` 且 `last_frame_at` 持续更新。
5. 触发测试风险事件，确认回调鉴权、JSON 解析、`event_id` 幂等和 `2xx` 响应。
6. 使用同一 `event_id` 模拟重复回调，确认不会重复保存或重复通知。
7. 更新一次直播地址，确认会话 ID 不变且继续出帧。
8. 停止会话并重复停止一次，确认接口幂等。

算法服务启动后，可通过 `${ALGORITHM_BASE_URL}/docs` 查看交互式接口文档，通过 `${ALGORITHM_BASE_URL}/openapi.json` 获取 OpenAPI 定义。实际联调以当前部署版本的 OpenAPI 和本文档标注的 `schema_version` 为准。
