# 算法设计与获奖竞争力审查报告

> 归档状态：2026-06-28 历史审查，代码行数、能力状态和待办均可能过期。当前入口见[项目文档](../../README.md)。

审查日期：2026-06-28

审查范围：本仓库算法原型研发部分，聚焦跌倒风险和心理健康预警两个模块。不评审产品化模块（家属端 App、社区后台、消息推送等）。

## 1. 结论先行

- **总体判断：中等偏强** — 算法设计和工程骨架在同类学生竞赛项目中属于较高水平，但从"能获奖"到"确实获奖"之间存在多个必须补齐的证据缺口。
- **获奖竞争力判断：中等** — 具备竞争基础，但当前缺少实验数据、训练模型和评估指标，答辩时可能被追问到无法回答的硬伤。
- **一句话理由：** 设计文档和工程骨架的完成度远高于实际算法验证和实验证据的完成度，存在"方案丰满、数据骨感"的风险。
- **最关键的 3 个风险：**
  1. **无训练模型、无评估指标、无消融实验** — 所有模块当前都是规则 baseline，没有一个模块完成了训练-评估-报告闭环。
  2. **自采数据为零** — `data/raw_videos/fall_risk/` 和 `data/annotations/fall_risk/` 下无实际数据文件，个体化基线、坐站、近跌倒模块的核心创新点缺乏训练和验证数据。
  3. **心理健康模块几乎空壳** — 仅有 ~110 行代码的加权求和规则，无研发计划、无数据源、无特征提取实现、无测试覆盖（仅 1 个 30 行测试），比赛答辩时无法支撑"多模态"和"心理健康"的题目承诺。

**免责声明：** 本报告基于仓库代码和文档证据给出技术判断，不保证获奖，也不代表评审委员会意见。挑战杯具体评审标准未在仓库中找到官方文本，以下按常见创新创业/科技竞赛评价维度推断。

## 4. 关键缺陷

按严重程度排序：

### 缺陷 1：无训练脚本、无评估脚本、无实验指标（严重度：致命）

- **问题：** `scripts/train/` 不存在，`scripts/evaluate/` 只有 README。研发计划中列出的 6 个训练任务（fall_event、near_fall、gait_risk、sit_stand_risk、baseline、fusion）全部未实现。无 Accuracy、Precision、Recall、F1、AUC、误报率、漏报率、提前预警时间等任何实验指标。
- **证据：** `ls scripts/train/` → No such file or directory；`scripts/evaluate/README.md` 仅 340 字节；当时的数据集方案文档明确承认"还没有完整的训练脚本和评估脚本"。新版数据集标注文档已拆出训练方案，避免与标注规范混写。
- **为什么影响：** 挑战杯评审几乎必然追问"你的模型准确率多少？"。规则 baseline 没有对照实验就无法证明算法有效性。所有文档中 `>= 90%` 高风险召回率、`>= 80%` 近跌倒 F1 的目标指标均无实测数据支撑。
- **建议修复方向：** 优先用公开数据（Fall Detection 2017, 997 Fall + 1017 ADL）训练 `fall_event_score` 二分类模型并报告指标。这是最快能产出"有数字的实验结果"的路径。

### 缺陷 2：自采数据为零（严重度：致命）

- **问题：** `data/raw_videos/fall_risk/` 为空目录。`data/annotations/fall_risk/` 下仅有 README，无 `action_labels.jsonl`、`event_labels.jsonl`、`risk_labels.jsonl`。`data/manifests/` 仅有 1 个 LE2I 样例 manifest (849 字节)。
- **证据：** `find data/raw_videos data/annotations/fall_risk -type f` 仅返回 `README.md`。
- **为什么影响：** 研发计划要求 8-12 人、15-20 类动作、600-1800 片段的推荐比赛规模。当前为 0。个体化基线需要 3-5 人连续 3-7 天数据，当前为 0。近跌倒检测需要模拟近跌倒正样本，当前为 0。没有自采数据，坐站、近跌倒、个体基线和最终融合模块的核心创新点将停留在"概念设计"层面。
- **建议修复方向：** 立即启动最小规模自采（3 人 × 8 类动作 × 5 次 = 120 片段），完成动作级和风险级标注。

### 缺陷 3：心理健康模块几乎空壳（严重度：高）

- **问题：** 心理健康模块总计 113 行 Python 代码（`pipeline.py` 77 行 + `features.py` 33 行 + `__init__.py` 3 行）。实现仅为 6 个预计算分数的加权求和规则。无数据采集方案、无特征提取实现、无数据源、无研发计划文档。测试仅 1 个（30 行），覆盖 1 个 happy path。
- **证据：** `docs/modules/mental_health/README.md` 第 1 行明确写道"本模块当前只建立算法工程边界，完整研发计划尚未编写"。第 31-38 行列出 4 项"需要补的前置文档"全部未完成。
- **为什么影响：** 项目标题是"跌倒风险、心理健康预警研究"。如果心理健康部分只有一个加权求和函数，评审会质疑"心理健康"是否只是凑题目。`mental_health_sample.json` 中 `social_withdrawal_score`、`negative_affect_score` 等字段的数据来源完全不清楚。
- **建议修复方向：** (1) 最低限度：补研发计划文档，明确数据源（活动节律来自视频、睡眠节律来自夜间活动检测、社交互动和语音/文本标记为"未来扩展"），明确哪些特征当前可从视频提取、哪些依赖外部输入。(2) 更好：实现活动节律和睡眠节律特征从视频/姿态输出中自动提取的链路。

### 缺陷 4：scene_risk_score 和 activity_rhythm_score 无上游生产者（严重度：高）

- **问题：** 融合层的 10 个输入特征中，`scene_risk_score` 和 `activity_rhythm_score` 在整个代码库中没有生产者实现。它们只在融合层被消费（权重分别为 0.08），但上游没有任何模块输出这两个分数。
- **证据：** `grep -r "scene_risk_score" src/` 仅在 `features.py`、`pipeline.py` 和 `baseline.py` 中出现作为输入字段，无任何计算逻辑。`grep -r "activity_rhythm_score" src/` 同样仅出现在消费端。
- **为什么影响：** 链路断裂。当前演示中这两个字段只能手动赋值或默认为 0。研发计划中"活动节律"和"场景风险"是重要特征组但无实现。
- **建议修复方向：** (1) 场景风险可通过 `scene_region` 字段的规则映射快速实现（床边 > 卫生间 > 走廊 > 客厅 > 其他，夜间加权）。(2) 活动节律需要按天聚合基线模块的输出。

### 缺陷 5：端到端离线演示脚本缺失（严重度：高）

- **问题：** 从视频到最终风险事件 JSON 的完整链路没有单一入口脚本。当前需要手动依次运行 7 个脚本（tracking → pose → pose_quality → gait → sit_stand → near_fall → baseline），再手动聚合结果并调用 `predict_from_features`。
- **证据：** `scripts/collect/` 有 7 个独立脚本，但无 `scripts/demo/run_end_to_end.py` 或类似入口。各脚本之间的输入输出路径需要手动衔接。
- **为什么影响：** 比赛演示时无法一键展示"视频进、风险出"的完整效果。评审可能质疑系统是否真正连通。
- **建议修复方向：** 写一个 `scripts/demo/run_fall_risk_pipeline.py`，接受视频路径，自动串联所有步骤，输出最终风险事件 JSON 和可视化。

### 缺陷 6：工具函数大量重复（严重度：低）

- **问题：** `_group_key`、`_record_sort_key`、`_record_time`、`_point`、`_has_point`、`_optional_number`、`_number_or_default`、`_clamp`、`_mean`、`_std`、`_ratio_score`、`_speeds`、`_path_max_deviation` 等 15+ 个工具函数在 `gait.py`、`sit_stand.py`、`near_fall.py`、`baseline.py` 中重复定义。
- **证据：** 代码审查发现 4 个文件中存在相同或近似的函数定义。
- **建议修复方向：** 抽取到 `src/elderly_monitoring/modules/fall_risk/_utils.py`。当前不影响功能但增加维护成本。

### 5.1 数据

- 权重固定（sum = 1.0），无校准依据。
- **判断：特征体系是概念占位，不是可运行实现。**

### 5.3 模型

- 加权求和 + 阈值分级。无机器学习模型。
- **判断：仅为规则占位。**

### 5.4 验证

- 1 个测试用例（`test_mental_health_pipeline.py`，30 行），仅测试高风险输入是否输出 3 级和 `manual_review`。
- 无负样本测试、无边界条件测试、无覆盖率测试。
- **判断：测试覆盖极度不足。**

- 文档中明确声明"不输出医学诊断"，`metadata.diagnosis = False` 在代码中强制设置。

### 5.7 总结

心理健康模块当前只是"接口占位 + 加权求和规则"。它不能支撑项目标题中"心理健康预警"的承诺。如果答辩时被追问"心理健康模块的数据来源和验证结果"，当前无法给出有说服力的回答。

## 9. 最终建议

### 当前是否建议直接参赛提交？

**不建议。** 当前最大硬伤是"无实验数据、无评估指标"。这不是一个可以用文档和代码架构弥补的缺陷。任何技术评审都会追问"你的模型效果如何？"，当前无法回答。

### 如果不能，必须先补什么？

**最低限度（3-5 天内必须完成）：**

1. 用公开数据训练 `fall_event_score` 二分类模型并报告 F1/AUC — 这是最快能产出"有数字的实验结果"的路径。
2. 完成端到端离线演示脚本 — 能一键从视频到风险 JSON。
3. 采集最小规模自采数据（至少 50 片段）— 证明不是纯公开数据拼接。

### 哪些内容可以作为答辩亮点？

1. **分层架构设计**：10 层固定路线从视频到风险输出，每层有明确的输入输出契约、质量控制和文档说明。这在学生项目中属于较高工程水平。
2. **可解释性设计**：`risk_factors` 和 `evidence_window` 输出、`confidence` 估计、低质量降级策略。不是黑盒模型。
3. **个体化基线概念**：相对个人历史而非群体阈值的偏离检测。概念有说服力，但需要数据验证。
4. **近跌倒前置预警定位**：从"跌倒后识别"升级为"跌倒前风险预测"的问题定义差异化。
5. **工程可复现性**：Conda 环境、58 个测试全通过、JSONL 数据契约、7 个采集脚本。

### 哪些内容答辩时不要夸大？

1. **不要说"已完成多模态融合"** — 当前主要是视觉单模态（视频 → 姿态 → 规则）。`activity_rhythm_score` 和 `scene_risk_score` 无生产者。GSTRIDE IMU 数据未接入融合管线。
2. **不要说"心理健康模块已实现"** — 当前是 113 行加权求和规则，无数据、无特征提取、无验证。
3. **不要说"已验证在真实老人场景中有效"** — 当前仅用 1 个 LE2I 公开视频跑过管线。无真实/半真实老人数据。
4. **不要把规则 baseline 的阈值说成"经过训练的模型参数"** — 所有权重和阈值都是工程经验值，未经标签校准。
5. **不要声称"高风险召回率 >= 90%"** — 这是研发计划的目标值，不是实测结果。当前无任何实测指标。
6. **不要说"已适配萤石开放平台"** — 当前仅保留了输入字段兼容，无 API 调用或 SDK 集成。
