## 二、项目定位

### 2.1 研究问题

本项目面向居家养老和独居老人安全监护场景，研究如何利用视频、人体姿态、行为节律和场景信息，识别老年人跌倒风险与心理健康风险的变化。

项目的重点不是只在老人已经跌倒后识别“发生了跌倒”，而是把监测时间点前移：

```text
跌倒前：识别步态变差、起身困难、近跌倒和行为基线异常
失衡中：识别踉跄、快速下沉、扶墙、急停恢复等近跌倒线索
跌倒后：识别疑似跌倒和长时间倒地，输出紧急风险事件
```

### 2.2 核心创新表达

当前建议统一使用以下表述：

> 个体化行为基线 + 近跌倒事件检测 + 可解释分级预警。

相较于单一跌倒动作识别，本方案强调三个变化：

| 传统方案 | 本项目方案 |
|---|---|
| 关注跌倒是否已经发生 | 同时关注跌倒前、失衡中和跌倒后 |
| 对所有老人使用相同阈值 | 将当前状态与个人历史基线比较 |
| 只输出类别或概率 | 输出等级、置信度、风险因子和证据时间窗 |
| 单帧或短片段判断 | 同时分析短时事件与长期趋势 |

### 2.3 项目边界

本仓库负责算法原型研发，主要包括：

- 数据读取、清洗、标注转换和特征提取。
- 人体检测、跟踪、姿态估计和行为分析。
- 跌倒风险与心理健康风险算法。
- 模型训练、推理、评估和标准化事件输出。

本仓库不负责：

- 家属端 App、社区端后台或完整可视化产品。
- 账号、权限、设备管理。
- 消息推送、电话通知、工单流转和线下处置。

需要注意的是，算法仓库不做业务系统，不代表比赛演示可以没有应用形态。萤石平台接入、风险结果展示和最小演示闭环需要由团队其他成员或单独工作流负责，并通过标准 JSON 与算法模块对接。

## 三、总体技术方案

### 3.1 跌倒风险固定算法路线

```text
萤石设备或开放平台视频流
  ↓
人体检测与跟踪
  ↓
人体姿态关键点提取
  ↓
关键点质量控制与时序平滑
  ↓
步态稳定性分析
  ↓
坐站转换能力分析
  ↓
近跌倒事件检测
  ↓
个体化行为基线建模
  ↓
轻量风险融合模型 + 规则校准
  ↓
跌倒风险等级 + 置信度 + 可解释风险因子 + 预警动作建议
```

这条路线包含三个硬约束：

1. 不从原始视频直接跳到最终风险结论，每一层保留明确输入、输出和质量信息。
2. 步态、坐站、近跌倒等局部模块只输出局部分数和证据，最终风险等级只由融合层产生。
3. 低质量视频或关键点缺失不能被直接解释为高风险，应降低置信度、限制分数或建议人工复核。

### 3.2 各层职责

| 层级 | 输入 | 主要处理 | 输出 |
|---|---|---|---|
| 检测与跟踪 | 视频帧 | 检测人体并保持跨帧轨迹 | 人体框、`track_id`、位置、速度、质量 |
| 姿态提取 | 视频帧/人体区域 | 提取肩、髋、膝、踝、腕等关键点 | 姿态 JSONL、关键点置信度 |
| 质量控制 | 原始姿态序列 | 过滤低置信度点、短缺失插值、平滑、跳变标记 | 稳定关键点与窗口质量 |
| 步态分析 | 稳定姿态序列 | 分析速度变化、对称性、横向摆动、停顿等 | `gait_risk_score` 与风险因子 |
| 坐站分析 | 稳定姿态序列 | 分析起身耗时、失败尝试、站稳时间、摇晃等 | `sit_stand_risk_score` 与事件证据 |
| 近跌倒检测 | 稳定姿态序列 | 检测横向失衡、快速下沉恢复、急停恢复等 | `near_fall_event_score` 与证据窗 |
| 个体基线 | 历史特征 + 当前特征 | 计算当前状态相对个人历史的偏离 | `baseline_deviation_score` |
| 风险融合 | 各局部分数、质量、场景和节律 | 规则评分卡或轻量模型融合 | 最终风险事件 JSON |

### 3.3 风险输出接口

算法统一输出事件化 JSON，业务或演示层只需要消费结果，不需要理解模型内部实现。

```json
{
  "module": "fall_risk",
  "device_id": "ezviz_cam_001",
  "person_id": "elder_001",
  "timestamp": "2026-07-11T14:20:00+08:00",
  "scene_region": "bedroom_bedside",
  "risk_level": 3,
  "risk_score": 0.84,
  "confidence": 0.87,
  "trigger_event": "near_fall",
  "risk_factors": [
    "重心快速横向偏移",
    "步态节律中断"
  ],
  "recommended_action": "notify_guardian",
  "evidence_window": {
    "start_time": 15.2,
    "end_time": 17.6
  },
  "model_version": "fall-risk-v0.1"
}
```

风险等级统一为 0-4 级：

| 等级 | 含义 | 算法建议动作 |
|---:|---|---|
| 0 | 正常 | 仅记录 |
| 1 | 低风险 | 持续观察 |
| 2 | 中风险 | 建议提醒老人或家属关注 |
| 3 | 高风险 | 建议主动通知家属 |
| 4 | 紧急风险 | 疑似跌倒或长时间倒地，建议紧急告警 |

算法只输出建议动作编码，不直接发送消息、拨打电话或创建工单。

### 3.4 心理健康风险路线

心理健康方向当前不做抑郁、焦虑等医学诊断，第一版建议采用更克制的路线：

```text
活动节律、睡眠节律、日常规律性等可观察特征
  ↓
个人历史基线
  ↓
异常偏离分数
  ↓
心理健康风险等级
  ↓
人工复核建议
```

候选特征包括活动量下降、房间移动减少、久坐久卧增加、夜间活动增加、生活规律波动等。问卷、语音和文本只作为数据可得后的增强方向，不能在尚无数据和验证时写成已实现能力。

### 4.2 当前成熟度判断

建议将当前阶段定义为：

> 分层算法原型已经搭建，单元级功能可运行；训练、实测指标、实时端到端演示和真实场景证据仍需补齐。

不能将当前状态表述为“完整系统已经实现”，原因包括：

- 当前步态、坐站、近跌倒、个体基线和风险融合主要是规则或统计 baseline。
- 尚无基于真实老人长期跌倒风险标签训练并校准的模型。
- 尚未形成规范的训练集、验证集和按人员隔离的盲测集结果。
- 当前没有可用于比赛材料的 Accuracy、Recall、F1、误报率和响应延迟实测表。
- 萤石开放平台尚未完成真实 API/SDK 接入验证。
- 实时视频上的自动滑窗、事件合并、节流和稳定性控制尚未形成完整主流程。
- 心理健康模块目前只有工程边界、候选特征和简单基线，成熟度明显低于跌倒风险模块。

### 4.3 已有方案的价值

虽然距离最终成果仍有明显缺口，但现有分层架构有三个直接价值：

1. 中间特征和质量信息可检查，便于定位误报来自检测、姿态、行为特征还是融合层。
2. 各模块通过结构化 JSONL/JSON 对接，可以由不同开发人员并行推进。
3. 规则 baseline 可作为后续 Logistic Regression、LightGBM、TCN 等模型的对照组，支持消融实验和可解释性分析。

## 五、当前主要问题与风险

### 5.1 最高优先级问题

| 优先级 | 问题 | 直接影响 | 建议动作 |
|---|---|---|---|
| P0 | 缺少实测指标 | 无法证明技术有效，比赛材料缺乏核心证据 | 尽快建立训练/验证/盲测流程并输出第一版指标 |
| P0 | 萤石平台尚未真实接入 | 与赛题平台要求和现场演示存在差距 | 立即确认账号、设备、视频流和开放平台权限 |
| P0 | 隐私合规材料不完整 | 自采数据存在合规风险，严重时影响参赛资格 | 自采前完成知情同意、脱敏、访问和退出删除规则 |
| P0 | 端到端链路未闭合 | 目前各脚本可运行，但还缺稳定的一键演示流程 | 完成视频到风险事件时间线的离线主流程 |
| P0 | 近跌倒误报证据不足 | 正常转身、下蹲、挥手可能触发误报 | 建立负样本集、误报表和阈值扫描实验 |

### 5.2 技术风险

- 2D 归一化坐标只能作为图像空间 proxy，不能直接解释为真实步速、距离或医学角度。
- 摄像头视角、低光、遮挡、下肢出画和跟踪 ID 切换会影响下游全部模块。
- 近跌倒事件样本少，直接训练复杂监督模型容易过拟合。
- 个体基线存在冷启动问题，新用户前几天没有稳定个人历史。
- 过早堆叠复杂模型会增加实时演示不稳定和实验不可复现的风险。

### 5.3 项目管理风险

- 跌倒风险和心理健康两条线同时全面推进，容易导致资源分散，两边都缺少可验证成果。
- 算法、数据、平台和比赛材料如果没有明确负责人，最终可能出现“代码有了但指标、演示或证明材料缺失”。
- 字段和阈值未经评审随意修改，会导致上下游脚本、样例和报告互相不一致。

## 六、后续研发方向与优先级

### 6.1 总体原则

下一阶段不以“增加模型名称”为目标，而以形成可验证证据为目标：

```text
可运行链路
  + 可复现数据
  + 明确指标
  + 误报/漏报分析
  + 萤石平台或设备演示
  = 可提交、可答辩的项目成果
```

### 6.2 推荐资源投入

在现有成熟度和比赛时间约束下，建议：

| 方向 | 建议投入 | 原因 |
|---|---:|---|
| 跌倒风险主链路、数据与评估 | 70% | 当前最接近形成可验证成果，也是核心技术主线 |
| 萤石接入、演示闭环与合规 | 20% | 属于比赛准入与落地证据，不能后置 |
| 心理健康最小可行方案 | 10% | 先完成数据可得性、非诊断边界和行为节律基线，避免无证据扩张 |

该比例是当前阶段建议，可根据团队人数调整，但不建议把两个算法方向按 1:1 平均分配。

### 6.3 第一阶段：未来 1-2 周

目标：从“模块可运行”推进到“端到端可验证”。

1. 完成视频到风险事件 JSON 的端到端离线脚本。
2. 固定第一版数据清单、人员划分规则和标注批次。
3. 建立正常动作负样本集，优先覆盖正常转身、坐下、下蹲、绕障、挥手、多人和遮挡。
4. 跑出规则 baseline 的第一版事件级指标，不等待复杂模型完成。
5. 集中管理阈值、模型版本和实验配置，保证结果可复现。
6. 启动萤石设备和开放平台权限确认，明确实时流获取方式。
7. 完成自采数据前的知情同意、脱敏和数据管理方案。

阶段交付：

- 一条可重复运行的离线演示命令。
- 一份数据版本与划分清单。
- 一张第一版指标表。
- 一份误报/漏报案例表。
- 一份萤石接入状态说明。

### 6.4 第二阶段：未来 3-5 周

目标：提高准确性并证明核心创新有效。

1. 对规则阈值进行验证集扫描，确定可解释基线。
2. 训练 Logistic Regression 或 LightGBM 轻量融合模型，与规则评分卡比较。
3. 对近跌倒尝试 Isolation Forest、One-Class SVM 等小样本异常检测方法。
4. 补充转身稳定性、轨迹连续性、场景和活动节律特征。
5. 建立个体基线冷启动策略，并比较“有/无个体基线”的指标变化。
6. 对 YOLOv8-pose 与 RTMPose 做关键点质量和下游误报对比，再决定是否切换主后端。

阶段交付：

- 规则 baseline 与轻量模型对比实验。
- 近跌倒、个体基线相关消融实验。
- 典型成功、误报和漏报案例。
- 推理延迟与关键点提取成功率报告。

### 6.5 第三阶段：提交前 2-3 周

目标：形成稳定演示和完整材料。

1. 完成萤石视频流或设备输入的实际联调。
2. 增加事件合并、节流、风险等级稳定和超时降级策略。
3. 固定模型、配置、数据版本和复现命令，停止无必要的大规模改动。
4. 完成独立盲测和最终指标报告。
5. 制作风险事件时间轴、个人趋势曲线、典型案例和消融实验图表。
6. 形成研究报告、技术文档、部署说明、测试报告和演示视频。

复杂深度模型只有在数据量、指标提升和推理预算同时满足时才进入演示主路径，否则保留为离线增强实验。

## 八、开发协作机制

### 8.1 任务拆分标准

每个开发任务至少写清楚：

- 背景：为什么要做。
- 范围：修改哪一层，不修改什么。
- 输入输出：涉及哪些字段、文件和下游消费者。
- 验收标准：什么结果算完成。
- 验证方式：需要跑哪些测试、烟测或评估。
- 负责人和截止时间。

推荐任务粒度为 0.5-2 个工作日。超过 3 天的任务应继续拆分，避免直到最后一天才暴露阻塞。

### 8.2 分支与提交建议

建议采用轻量分支协作：

```text
main                    可运行、可演示的稳定分支
feature/<模块>-<目标>    新功能
fix/<模块>-<问题>        缺陷修复
experiment/<主题>       不保证进入主线的实验
docs/<主题>             文档与材料
```

每次提交只解决一个明确问题，提交信息示例：

```text
feat(near-fall): 增加正常转身排除特征
fix(pose-quality): 修复短缺失插值边界
docs(data): 更新第一批标注数据清单
```

实验分支的结论应落到实验记录中，不能只留下代码。未证明有效的复杂模型不直接替换稳定主线。

### 8.3 合并前检查

代码合并前至少满足：

1. 明确修改的是算法路线中的哪一层。
2. 新增或修复行为有对应测试；缺陷优先先补回归测试。
3. 使用项目指定环境运行相关测试。
4. 涉及检测、跟踪或姿态时，运行真实视频烟测。
5. 修改字段、阈值或行为时，同步更新配置、样例和说明文档。
6. 记录已知限制，不把未验证能力描述为已经完成。
7. 不在代码、日志、样例和提交记录中写入真实个人身份信息。

项目统一验证命令：

```bash
conda run -n eldercare-ai python -m pytest -q
```

### 8.4 Definition of Done

一个算法任务只有同时满足以下条件才算完成：

- 功能或实验目标已经实现。
- 输入输出契约清楚，未破坏下游兼容性。
- 相关测试通过，必要的真实视频烟测通过。
- 有最小可复现样例或命令。
- 指标、运行结果或案例已经记录。
- 文档说明与实际代码一致。
- 已知限制和失败样本已经登记。

“代码写完”不等于“任务完成”。

### 8.5 沟通节奏

建议采用以下节奏：

| 频率 | 形式 | 重点 |
|---|---|---|
| 每日 | 群内简短同步 | 昨日完成、今日计划、阻塞事项 |
| 每周一次 | 30-45 分钟技术例会 | 演示可运行结果、看指标和失败样本、调整优先级 |
| 每个里程碑 | 评审会议 | 检查数据、代码、指标、演示和材料是否同时完成 |
| 重大接口变更前 | 小范围设计评审 | 确认字段、兼容性、迁移方案和测试范围 |

周会不建议只汇报“做了什么”，应至少展示一项证据：测试结果、指标表、样例输出、失败案例或可运行演示。

### 8.6 文档与实验记录

统一记录以下内容：

- 数据来源、许可、版本和划分。
- 配置、模型版本、随机种子和运行环境。
- 指标定义、阈值和计算脚本。
- 成功案例、误报案例、漏报案例和低质量案例。
- 实验结论：保留、放弃或继续验证，以及原因。

避免出现同一个指标在不同汇报中口径不同，或模型更新后报告仍使用旧结果的情况。

## 附录 B：常见问题与回答口径

### Q1：为什么不直接训练一个视频分类模型？

直接分类虽然实现快，但难以区分误报来源，也很难解释为什么判为高风险。本项目保留检测、姿态质量、步态、坐站、近跌倒和个体基线等中间层，便于解释、调试、消融和后续替换模型。

### Q2：为什么第一版使用规则 baseline？

当前标签规模有限，规则 baseline 更容易运行、复现和解释，也能作为后续轻量模型的对照。规则不是最终目标，但在没有可靠指标前直接换复杂模型并不能证明效果更好。

### Q3：公开跌倒数据能否证明长期跌倒风险？

不能。公开跌倒数据主要用于验证检测、姿态、动作和事件链路。长期风险、活动节律和个体基线需要长期观测数据或更合适的数据集支撑，两类证据必须分开说明。

### Q4：心理健康风险如何避免变成医学诊断？

只根据可观察的活动和生活节律变化输出风险提示，并建议人工复核。算法不直接输出抑郁症、焦虑症等诊断结论，高风险结果需要结合问卷或专业人员评估。

### Q5：团队下一步最重要的工作是什么？

不是继续扩充模型清单，而是跑通端到端链路、固定数据与评估口径、产出第一版实测指标、分析误报漏报，并尽快落实萤石接入和合规要求。
